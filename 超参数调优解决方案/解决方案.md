# RAG 超参数调优解决方案

## 问题描述（原文）

> 超参数（chunk size、top-K、overlap）调优依赖经验，缺乏理论指导

在 RAG 系统中，chunk size（文本切块大小）、top-K（检索返回数量）和 overlap（切块重叠度）是影响检索效果的关键超参数。目前业界普遍采用经验值或手动调参，缺乏系统化的理论指导框架，导致不同场景下参数设置不合理，检索性能大幅波动。

---

## 解决方案

### 方案一：AutoRAG 自动化超参数搜索框架

**核心思路**

将 RAG 超参数优化问题形式化为自动化搜索问题，通过模块化流水线逐步贪心地寻找各模块的最优参数组合，无需人工介入。

**实现方法**

1. 将 RAG pipeline 拆分为独立模块节点（查询扩展 → 检索 → 段落增强 → 重排序 → 生成）
2. 为每个节点定义候选超参数搜索空间（如 chunk_size ∈ {128, 256, 512, 1024}）
3. 使用贪心算法，在每个节点上通过评估指标（NDCG、MRR、EM 等）选择当前最优配置
4. 支持 chunk size、top-K、overlap、embedding 模型等多维超参数的联合优化

**推荐工具**

- [AutoRAG](https://github.com/Marker-Inc-Korea/AutoRAG) — 开源框架，支持完整 RAG pipeline 超参数自动优化
- 配套评估数据集格式：QA pairs + ground truth

**参考来源**

- [AutoRAG: Automated Framework for Optimization of Retrieval Augmented Generation Pipeline (arxiv 2410.20878)](https://arxiv.org/html/2410.20878v1)
- [AutoRAG LlamaIndex 集成文档](https://docs.llamaindex.ai/en/stable/examples/param_optimizer/param_optimizer/)

### 方案二：AutoRAG-HP — 在线多臂老虎机（MAB）超参数调优

**核心思路**

将 RAG 超参数选择建模为一个**在线多臂老虎机（Multi-Armed Bandit, MAB）**问题，通过分层 MAB 结构在大搜索空间中高效探索，相比网格搜索可节省约 80% 的 LLM API 调用。

**实现方法**

1. **高层 MAB**：在模块级别（如不同检索策略、不同 top-K 范围）做选择
2. **低层 MAB**：在每个模块内部的具体超参数值之间做选择
3. 在线学习：每次新查询到来时，基于历史奖励信号（检索准确率）更新 MAB 策略
4. 支持动态场景下超参数的持续自适应调整

**适用场景**

- 数据分布会随时间变化的动态 RAG 系统
- 不希望离线大规模实验、需要在线实时调整的场景

**参考来源**

- [AutoRAG-HP: Automatic Online Hyper-Parameter Tuning for Retrieval-Augmented Generation (ACL 2024)](https://aclanthology.org/2024.findings-emnlp.223/)
- [AutoRAG-HP arxiv 预印本](https://arxiv.org/abs/2406.19251)

### 方案三：贝叶斯优化（Bayesian Optimization）

**核心思路**

用贝叶斯优化替代网格搜索，以更少的实验次数找到超参数最优点。利用高斯过程代理模型（Gaussian Process Surrogate）对参数空间建模，通过采集函数（Acquisition Function）智能决定下一次实验的参数配置。

**实现方法**

1. 定义超参数搜索空间：`chunk_size ∈ [64, 1024]`，`top_k ∈ [1, 20]`，`overlap_ratio ∈ [0, 0.5]`
2. 构建评估函数：给定一组超参数，运行 RAG pipeline 并计算评估指标（如 Recall@K、Answer Correctness）
3. 使用 Optuna / Ax / BoTorch 库运行贝叶斯优化循环（通常 50-200 次实验即可收敛）
4. 多目标扩展：使用 qLogNEHVI 采集函数同时优化检索质量和推理延迟

**推荐工具**

- [Optuna](https://optuna.org/) — 轻量级超参数优化框架
- [BoTorch](https://botorch.org/) — Facebook 出品，支持多目标贝叶斯优化
- [Kubeflow Katib](https://blog.kubeflow.org/katib/rag/) — Kubernetes 原生超参数调优平台

**参考来源**

- [Beyond Grid Search: Leveraging Bayesian Optimization for RAG (EACL 2026 Industry Track)](https://aclanthology.org/2026.eacl-industry.19.pdf)
- [Leveraging Bayesian Optimization for Accelerating RAG Pipeline (TUM 硕士论文)](https://www.cs.cit.tum.de/fileadmin/w00cfj/sebis/_my_direct_uploads/Xueru_Zheng_MT.pdf)
- [Multi-Objective Hyperparameter Optimization for LLM and RAG Systems (arxiv 2502.18635)](https://arxiv.org/html/2502.18635v1)

### 方案四：多尺度分层检索（Multi-Scale Chunking + RRF 融合）

**核心思路**

不再纠结"最优 chunk size"，而是同时对多个 chunk size 建立索引，查询时并行检索后通过**倒序排名融合（Reciprocal Rank Fusion, RRF）**合并结果，从而规避单一 chunk size 选择的局限性。

**实现方法**

1. 对同一语料库建立多套索引（如 chunk_size ∈ {100, 200, 500} tokens）
2. 查询时并行在所有索引上检索，获得多份候选列表
3. 使用 RRF 公式 `score(d) = Σ 1/(k + rank_i(d))` 合并排名
4. 实验表明此方法可在不重训模型的情况下提升 1–37% 的检索性能

**参考来源**

- [Rethinking Chunk Size for Long-Document Retrieval: A Multi-Dataset Analysis (arxiv 2505.21700)](https://arxiv.org/abs/2505.21700)
- [Chunk size is query-dependent: a simple multi-scale approach to RAG retrieval (AI21)](https://www.ai21.com/blog/query-dependent-chunking/)

### 方案五：基于任务类型的经验性理论指导框架

**核心思路**

通过系统性实验研究建立"任务类型 → 超参数范围"的映射关系，提供有理论支撑的经验基线，作为调参起点。

**实现方法**

基于现有研究综合得出的参数指导原则：

| 任务类型 | 推荐 chunk_size | 推荐 top-K | overlap 比例 |
|---------|----------------|-----------|------------|
| 精确事实型 QA | 64–256 tokens | 3–5 | 10–15% |
| 需要上下文的理解型 QA | 512–1024 tokens | 5–10 | 15–20% |
| 长文档摘要/多跳推理 | 1024–2048 tokens | 10–20 | 20–30% |
| 混合型（通用场景） | 512 tokens（起点） | 5 | 10% |

**注意事项**

- embedding 模型对 chunk size 敏感性不同（Stella 偏好大块，Snowflake 偏好小块），需结合所用模型调整
- 建议先用上表经验值作为起点，再结合方案一/三进一步优化

**参考来源**

- [NVIDIA Blog: Finding the Best Chunking Strategy for Accurate AI Responses](https://developer.nvidia.com/blog/finding-the-best-chunking-strategy-for-accurate-ai-responses/)
- [Weaviate Blog: Chunking Strategies for RAG](https://weaviate.io/blog/chunking-strategies-for-rag)
- [Optimizing Retrieval-Augmented Generation: Analysis of Hyperparameter Impact (arxiv 2505.08445)](https://arxiv.org/html/2505.08445)
