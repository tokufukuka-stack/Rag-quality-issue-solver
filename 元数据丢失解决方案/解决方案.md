# 元数据丢失解决方案

## 问题描述（原文）

> 元数据丢失——文档标题、作者、日期等结构化信息未被保留

在 RAG 系统中，文档在解析、切块（chunking）和向量化过程中，常常丢失原始文档携带的结构化元信息（如标题、作者、创建日期、文档类型、章节层级等）。这导致检索时无法利用这些信息进行过滤与排序，生成结果也无法正确引用来源，严重影响 RAG 系统的精度与可信度。

---

## 主要解决方案

### 方案一：文档加载阶段的元数据自动提取与保留

**核心思路：**
在文档解析（ingestion）阶段，使用支持结构化元数据提取的解析工具，将文件级元数据（标题、作者、日期、URL、文件名、页码等）自动附加到每个 Document 对象或 Node 对象上，并在切块时随 chunk 传播保留。

**实现方法：**
- 使用 **LangChain** 的 `Document` 对象：每个文档包含 `page_content`（正文）和 `metadata`（字典，可存储任意键值对）。可通过自定义 `Document Loader` 在加载时填充元数据字段。
- 使用 **LlamaIndex** 的 `Document` 和 `Node` 对象：支持 `metadata` 字段，并提供 `MetadataExtractor` 管道（自动提取关键词、摘要、实体等）。在 `SentenceSplitter` 切块时，`metadata` 默认继承到每个子节点。
- 使用 **Unstructured.io**：专为非结构化文档解析设计，能从 PDF、Word、HTML 等格式中提取并规范化元数据字段（包括文档标题、作者、创建时间等），输出标准化的结构化 JSON。

**推荐工具/框架：**
- LangChain Document Loaders（`PyPDFLoader`, `UnstructuredFileLoader` 等）
- LlamaIndex `SimpleDirectoryReader` + `MetadataExtractor`
- Unstructured.io（开源版或 API）

**参考来源：**
- [LlamaIndex Loading Data Documentation](https://developers.llamaindex.ai/python/framework/understanding/rag/loading/)
- [Unstructured.io: Metadata for RAG](https://unstructured.io/insights/how-to-use-metadata-in-rag-for-better-contextual-results)
- [Databricks RAG Data Pipeline Best Practices](https://docs.databricks.com/aws/en/generative-ai/tutorials/ai-cookbook/quality-data-pipeline-rag)

### 方案二：LLM 辅助元数据增强提取（LLM-Based Metadata Extraction）

**核心思路：**
对于无法从文件属性中直接获得的隐性元数据（如主题类别、文档摘要、关键实体、领域标签），使用 LLM 在 ingestion 阶段对每个 chunk 或整篇文档进行自动抽取，并将提取结果作为结构化元数据存储。

**实现方法：**
- 在 LlamaIndex 中配置 `MetadataExtractor`，使用预置的提取器（如 `TitleExtractor`, `KeywordExtractor`, `SummaryExtractor`, `QuestionsAnsweredExtractor`），对文档或 chunk 调用 LLM 提取对应字段。
- 在 LangChain 中，可借助 `LLMChain` 或 `ChatPromptTemplate` 设计提取 Prompt，批量处理 Document 并将结果写回 `metadata` 字典。
- 参考 **MDKeyChunker** 框架（arXiv 2025）：通过滚动关键词传播策略（Rolling Keys）解决跨 chunk 元数据不一致的问题，实现高精度的 chunk 级元数据标注。

**推荐工具/框架：**
- LlamaIndex `MetadataExtractor` pipeline
- LangChain `LLMMetadataExtractor`（自定义实现）
- MDKeyChunker（arXiv 研究原型）

**参考来源：**
- [Advanced RAG techniques with LangChain — Part 7 (Medium)](https://medium.com/@roberto.g.infante/advanced-rag-techniques-with-langchain-part-7-843ecd3199f0)
- [MDKeyChunker: Single-Call LLM Enrichment (arXiv 2025)](https://arxiv.org/html/2603.23533)
- [deepset: Leveraging Metadata in RAG Customization](https://www.deepset.ai/blog/leveraging-metadata-in-rag-customization)

### 方案三：元数据过滤增强检索（Metadata-Filtered Retrieval）

**核心思路：**
将保留的元数据字段用于检索阶段的预过滤（pre-filtering）或后过滤（post-filtering），在向量相似度搜索之前先按元数据条件（如日期范围、作者、文档类型）缩小候选集，从而提升检索精度和速度。

**实现方法：**
- 在向量数据库（如 **Pinecone**, **Weaviate**, **Qdrant**, **ChromaDB**, **pgvector**）中，将元数据字段设置为过滤字段（filter fields），在查询时传入过滤条件。
- 使用 **Haystack** 的 `MetadataRouter` 或查询时元数据提取（query-time metadata extraction）：先用 LLM 从用户问题中解析出隐含的元数据限制（如"2023年发布的论文"），再将其转化为过滤条件。
- **Multi-Meta-RAG**（arXiv 2024）：利用 LLM 提取多维度元数据并存入数据库，检索时基于元数据过滤大幅提升多跳问题的 MRR@10、MAP@10 等指标。

**推荐工具/框架：**
- Pinecone / Weaviate / Qdrant（原生支持 metadata filtering）
- Haystack `MetadataRouter`
- LlamaIndex `MetadataFilters`

**参考来源：**
- [Multi-Meta-RAG: Improving RAG using Database Filtering with LLM-Extracted Metadata (arXiv 2024)](https://arxiv.org/abs/2406.13213)
- [Haystack: Advanced Retrieval — Extract Metadata from Queries](https://haystack.deepset.ai/blog/extracting-metadata-filter)
- [CodeSignal: Metadata-Based Filtering in RAG Systems](https://codesignal.com/learn/courses/scaling-up-rag-with-vector-databases/lessons/metadata-based-filtering-in-rag-systems)

### 方案四：结构化元数据与向量混合嵌入（Hybrid Metadata Embedding）

**核心思路：**
将元数据信息融合进向量表示中（而非仅作为过滤字段），通过在 chunk 文本前缀拼接元数据（如"[Title: xxx] [Author: yyy] [Date: zzz]"）或使用双编码器（dual-encoder）分别嵌入内容与元数据再合并，使元数据语义影响向量相似度计算。

**实现方法：**
- **前缀拼接法（Prefix Prepending）**：在 chunk 文本的开头追加格式化的元数据字段，再整体送入 embedding 模型。简单有效，适用于大多数 embedding 模型。
- **双编码器（Dual Encoder）**：参考 arXiv 2026 论文"Utilizing Metadata for Better RAG"，独立嵌入内容和元数据，最终合并为统一向量，与普通文本基线相比检索准确率显著提升。
- **SRAG（Structured RAG，arXiv 2026）**：在 chunk 上附加结构化信息（主题、情感、知识图谱三元组、语义标签），增强文档内聚性，减少跨文档混淆。

**推荐工具/框架：**
- 任意 embedding 模型（前缀拼接法无需额外框架）
- Sentence Transformers（可微调支持元数据嵌入）
- LlamaIndex `NodeParser` 的 `include_metadata` 选项

**参考来源：**
- [Utilizing Metadata for Better Retrieval-Augmented Generation (arXiv 2026)](https://arxiv.org/html/2601.11863v1)
- [SRAG: RAG with Structured Data Improves Vector Retrieval (arXiv 2026)](https://arxiv.org/html/2603.26670)
- [Metadata-Driven RAG for Financial QA (arXiv 2025)](https://arxiv.org/html/2510.24402v1)

### 方案五：元数据规范化与统一 Schema 管理

**核心思路：**
在多来源、多格式文档的场景下，制定统一的元数据 Schema（如 Dublin Core、自定义业务 Schema），并在 ingestion 管道中强制校验和标准化元数据字段，确保所有入库文档的元数据完整性和一致性。

**实现方法：**
- 定义元数据 Schema：包含必填字段（source, doc_id, created_at）和可选字段（author, title, section, page_num, doc_type 等），使用 Pydantic 模型或 JSON Schema 进行校验。
- 在 ingestion 管道中增加元数据补全步骤：对缺失字段尝试从文件系统（文件名、修改时间）、文档内容（正则匹配标题行）、外部 API（DOI 查询论文信息）等多个来源补全。
- 使用 **Databricks** 的 RAG 数据质量管道：原始数据保留在源表，解析后的结构化数据（含元数据）存储在目标表，确保可追溯性和审计能力。
- 定期检查索引中的元数据完整率，设置告警阈值。

**推荐工具/框架：**
- Pydantic（元数据 Schema 定义与校验）
- Apache Spark / Databricks（大规模 ingestion 管道）
- LangChain `DocumentTransformer`（批量补全元数据）
- LlamaIndex `TransformComponent`
