# 多租户隔离不足解决方案

## 问题描述（原文）

**多租户隔离不足——不同用户的数据在检索时可能交叉污染**

在多租户 RAG 系统中，多个用户或组织共享同一检索基础设施时，若缺乏严格的隔离机制，一个租户的查询可能检索到属于其他租户的文档片段（chunk），导致数据泄露、隐私侵犯及系统信任度下降。这在 SaaS 场景、企业级知识库中尤为危险。

---

## 主要解决方案

### 方案一：Silo 模式（孤岛隔离）—— 每租户独立索引

**核心思路**
为每个租户创建完全独立的向量数据库索引（collection/index），从物理层面实现租户隔离，彻底消除跨租户检索的可能性。

**实现方法**
- 每个新租户入驻时，在向量数据库（如 Milvus、Weaviate、Pinecone）中创建独立的 collection 或 index
- 文档嵌入（embedding）写入时，路由到该租户专属的存储空间
- 查询时根据 JWT token 或 API Key 中携带的 `tenant_id`，将请求路由到对应 collection
- Weaviate 原生支持多租户架构，每个租户拥有独立分片（shard），做到物理级隔离

**推荐工具/框架**
- [Weaviate Multi-Tenancy](https://weaviate.io/blog/weaviate-multi-tenancy-architecture-explained)（原生支持，每租户独立分片）
- [Milvus Database-level 或 Collection-level 多租户](https://milvus.io/blog/build-multi-tenancy-rag-with-milvus-best-practices-part-one.md)

**参考来源**
- [Silo, Pool, and Bridge for Multi-Tenant RAG — IJETCSIT](https://www.ijetcsit.org/index.php/ijetcsit/article/view/551)
- [Rethinking Vector Search at Scale: Weaviate's Native Multi-Tenancy](https://weaviate.io/blog/weaviate-multi-tenancy-architecture-explained)

### 方案二：Pool 模式（共享索引 + 元数据过滤）

**核心思路**
所有租户的数据存储在同一个向量索引中，但每条向量记录附带 `tenant_id` 元数据字段，查询时在相似度检索的同时强制施加租户过滤条件（pre-filter 或 post-filter），确保返回结果只属于当前租户。

**实现方法**
- 文档入库时，在每个 chunk 的 metadata 中注入 `tenant_id` 字段
- 向量检索时使用如下形式的过滤条件：
  ```
  WHERE tenant_id = 'current_tenant_id'
  ```
- Pinecone 通过 **namespace** 机制实现逻辑隔离，查询时必须指定 namespace，跨 namespace 访问被严格拒绝
- 需要确保过滤条件在查询层强制执行，避免查询层 bypass

**推荐工具/框架**
- [Pinecone Namespaces](https://www.pinecone.io/learn/series/vector-databases-in-production-for-busy-engineers/vector-database-multi-tenancy/)
- [Amazon OpenSearch 索引级隔离](https://aws.amazon.com/blogs/machine-learning/multi-tenant-rag-with-amazon-bedrock-knowledge-bases/)

**参考来源**
- [Multi-Tenancy in Vector Databases — Pinecone](https://www.pinecone.io/learn/series/vector-databases-in-production-for-busy-engineers/vector-database-multi-tenancy/)
- [Building Multi-Tenant RAG Applications With PostgreSQL — TigerData](https://www.tigerdata.com/blog/building-multi-tenant-rag-applications-with-postgresql-choosing-the-right-approach)

### 方案三：JWT + 细粒度访问控制（FGAC）

**核心思路**
在身份认证层引入 JWT（JSON Web Token）或 OIDC/SAML，将 `tenant_id` 等权限信息嵌入 token，API Gateway 解码后将租户上下文注入到 RAG 管道的每个阶段，配合细粒度访问控制（Fine-Grained Access Control）实现文档级、chunk 级的权限校验。

**实现方法**
1. 用户通过企业 IdP（OIDC/SAML）认证，获取包含 `tenant_id` claim 的 JWT
2. API Gateway 验证 token，提取 `tenant_id` 传递给后端服务
3. 检索服务在构建查询时，自动附加 `tenant_id` 过滤，不依赖调用方自觉传参
4. 行级安全（Row-Level Security）在数据库层面强制隔离，即使查询层出现 bug 也无法泄露跨租户数据
5. 结合 Amazon Bedrock 的 Knowledge Bases，可为每个租户配置独立的数据源权限

**推荐工具/框架**
- Amazon OpenSearch + Bedrock（JWT + FGAC）
- Azure AI Search + Azure AD（RBAC + 文档级权限）
- PostgreSQL Row-Level Security（pgvector 扩展）

**参考来源**
- [Multi-tenant RAG with Amazon Bedrock and OpenSearch using JWT — AWS](https://aws.amazon.com/blogs/machine-learning/multi-tenant-rag-implementation-with-amazon-bedrock-and-amazon-opensearch-service-for-saas-using-jwt/)
- [Design a Secure Multitenant RAG Inferencing Solution — Microsoft Azure](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/secure-multitenant-rag)

### 方案四：Curator 算法 —— 多租户向量索引的高效隔离

**核心思路**
学术界提出的 Curator 方法，专门针对多租户向量数据库中的索引效率和隔离问题。通过构建租户感知（tenant-aware）的索引结构，在保持高效检索性能的同时，从算法层面防止跨租户数据访问。

**实现方法**
- Curator 在索引构建阶段将租户信息编码进图结构（如 HNSW），使查询时的图遍历天然被限制在租户边界内
- 相比纯过滤方案，Curator 避免了 post-filter 导致的召回率下降问题
- 适合数据量巨大、租户数量众多的生产环境

**参考来源**
- [Curator: Efficient Indexing for Multi-Tenant Vector Databases — arXiv](https://arxiv.org/html/2401.07119v1)

### 方案五：全链路租户上下文强制传播（Pipeline 级防护）

**核心思路**
仅在存储层做隔离是不够的，需要在 RAG 管道的每个阶段（摄入 → 切片 → 嵌入 → 索引 → 检索 → 生成）都强制携带并校验租户上下文，同时收紧日志和可观测性管道，防止日志中出现跨租户信息泄露。

**实现方法**
1. 在摄入（Ingest）阶段：文档入库时绑定 `tenant_id`，拒绝无租户标识的写入
2. 在检索（Retrieve）阶段：中间件层强制注入 `tenant_id` 过滤，不允许绕过
3. 在生成（Generate）阶段：LLM 调用的 prompt 中，将上下文限定为当前租户检索到的内容
4. 日志脱敏：审计日志中对 embedding vector、文档内容进行脱敏处理，防止通过日志推断其他租户数据
5. 定期审计 namespace/partition 隔离规则的有效性
