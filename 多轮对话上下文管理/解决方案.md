# 多轮对话中上下文管理复杂问题解决方案

## 问题描述（原文）

**多轮对话中上下文管理复杂，历史信息与检索结果如何平衡**

在 RAG 系统的多轮对话场景中，随着对话轮数增加，系统需要同时管理：
- 当前用户提问
- 历史对话记录（可能跨越数十轮）
- 每轮新检索的文档片段

这三者共同竞争有限的上下文窗口空间。若历史信息占比过大，检索结果被压缩；若过度强调检索，历史语义丢失导致指代消歧失败（如用户说"它"指代前几轮提到的概念）。如何在两者之间动态权衡，是多轮对话 RAG 的核心工程挑战。

---

## 主要解决方案

### 方案一：对话历史压缩 + 摘要记忆（Conversation Summary Memory）

**核心思路**

不将完整历史对话逐字放入上下文，而是使用一个轻量 LLM（或规则方法）将历史对话实时压缩为"摘要"，仅保留关键信息。新的检索查询基于"摘要 + 当前问题"构建，而非"全量历史 + 当前问题"。

**实现方法**

1. 每完成一轮对话，调用摘要模型将历史记录压缩为 200-400 tokens 的摘要。
2. 摘要存储在 `ConversationSummaryMemory`（LangChain 内置组件）或自定义存储中。
3. 构建检索查询时，拼接：`[摘要] + [当前问题]` 形成 standalone query。
4. 检索结果与摘要一同注入生成 prompt。

**推荐工具/框架**

- LangChain: `ConversationSummaryMemory`, `ConversationSummaryBufferMemory`
- LlamaIndex: `SummaryMemory`
- 摘要模型：`gpt-4o-mini`、`claude-haiku` 等低成本模型

### 方案二：查询改写（Query Rewriting / Standalone Question Generation）

**核心思路**

多轮对话中用户的提问往往包含指代（如"它"、"这个"、"上面提到的"），直接用原始问题做向量检索会导致召回失败。在检索前，先用 LLM 将含指代的问题改写为自包含的独立问题（standalone question），再送入检索器。

**实现方法**

1. 维护一个滑动窗口（通常最近 3-5 轮）的对话历史。
2. 在检索前，调用 LLM 将当前问题 + 近期历史 → 改写为独立问题，如：
   - 原问题：`"它的参数量是多少？"`
   - 改写后：`"LLaMA 3 模型的参数量是多少？"`
3. 用改写后的问题做向量检索，再将检索结果 + 原始对话历史注入生成 prompt。

**推荐工具/框架**

- LangChain: `create_history_aware_retriever`（内置查询改写链）
- LlamaIndex: `Query Transform Cookbook`，包含 `HyDE`、`StepDecompose`、`ReAct` 改写策略
- 官方文档参考：[LlamaIndex Query Transform Cookbook](https://developers.llamaindex.ai/python/examples/query_transformations/query_transform_cookbook/)

**参考来源**

- [Contextual RAG: Maintaining Conversation Context in Retrieval - ChatNexus](https://articles.chatnexus.io/knowledge-base/contextual-rag-maintaining-conversation-context-in-retrieval/)

### 方案三：多粒度记忆分层架构（Multi-Granularity Memory）

**核心思路**

不同的查询类型对历史信息的粒度需求不同：时序推理问题需要会话级摘要，知识更新类问题需要轮次级细节，偏好类问题需要高度压缩的用户画像。通过构建多级记忆层（Turn-level / Session-level / Summary-level），配合一个"粒度路由器"（Granularity Router）动态选择最合适的记忆层注入检索。

**实现方法**

1. 存储三层记忆：
   - **Turn-level**：原始的对话轮次记录（细粒度）
   - **Session-level**：一个会话的整体摘要
   - **Summary-level**：跨会话的用户长期偏好摘要
2. 每次查询时，粒度路由器（可以是小型分类器或 LLM prompt）判断当前问题属于哪类，选择对应记忆层。
3. 将选中记忆层内容 + 检索结果 + 当前问题一同注入生成。

**推荐工具/框架**

- Mem0（开源记忆管理库）：支持多层记忆操作（ADD/UPDATE/DELETE/NOOP）
- Zep：面向对话 AI 的长期记忆管理平台
- MemOS（Memory OS）：系统级记忆管理框架

**参考来源**

### 方案四：动态上下文窗口管理与语义分块（Adaptive Context Window）

**核心思路**

上下文窗口是有限资源，不应静态分配给历史和检索结果各自固定比例，而应根据当前查询的复杂度和对历史依赖程度动态调整分配比例。同时，对历史对话按语义边界（而非固定轮次）切分，确保注入的历史片段是语义完整的单元。

**实现方法**

1. 使用**查询复杂度评估器**（可以是简单的 prompt 分类器）判断当前问题是否依赖历史信息。
2. 根据评估结果动态分配 token budget：
   - 高历史依赖 → 增大历史摘要比例，减少检索片段数量
   - 低历史依赖 → 减少历史占用，放入更多检索结果
3. 对检索结果进行**语境压缩**（Contextual Compression）：仅保留与当前问题最相关的句子，去除冗余文本。
4. LangChain 的 `ContextualCompressionRetriever` 可直接实现步骤3。

**推荐工具/框架**

- LangChain: `ContextualCompressionRetriever`, `LLMChainFilter`, `EmbeddingsFilter`
- LlamaIndex: Long Context RAG 架构（支持动态 top-k 调整）
- 参考：[Long Context RAG: New Architectures and Tradeoffs - LlamaIndex](https://www.llamaindex.ai/blog/towards-long-context-rag)

**参考来源**

- [The Context Collapse Crisis: Why Multi-Turn RAG Loses Track - RAG About It](https://ragaboutit.com/the-context-collapse-crisis-why-your-multi-turn-rag-system-loses-track-after-5-questions/)

### 方案五：Memory OS / 外部记忆管理系统

**核心思路**

将对话记忆彻底从上下文窗口中解耦，使用专门的外部记忆管理系统（如 Mem0、MemOS、Zep）持久化存储和检索历史信息。系统在每轮对话时动态检索"相关历史记忆"，与当前文档检索结果并行注入，而非依赖线性的上下文追加。

**实现方法**

1. 每轮对话结束后，调用记忆管理器提取关键信息并存储到向量记忆库。
2. Memory Manager 执行 ADD（新增）/ UPDATE（更新已有记忆）/ DELETE（删除过时记忆）/ NOOP（无操作）四种操作，维持记忆状态演进。
3. 下一轮对话时，先用当前问题检索外部记忆库，获取"相关历史记忆片段"。
4. 将记忆片段 + 文档检索结果 + 当前问题一并注入生成 prompt。
5. 记忆库独立于上下文窗口，可无限扩展。

**推荐工具/框架**

- [Mem0](https://github.com/mem0ai/mem0)（开源）：专为 AI Agent 设计的记忆层
- [Zep](https://github.com/getzep/zep)（开源/商业）：对话 AI 长期记忆平台
- MemOS（Memory OS for AI System）：系统级记忆操作系统
