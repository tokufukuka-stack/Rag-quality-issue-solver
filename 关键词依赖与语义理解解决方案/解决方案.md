# 关键词依赖过重——无法理解同义词、隐喻等语言变体

## 问题描述（原文）

> 关键词依赖过重，无法理解同义词、隐喻等语言变体

传统 RAG 系统在检索阶段往往依赖关键词匹配（如 BM25、TF-IDF），当用户查询使用了与索引文档不同的表述方式——例如同义词（"residence" vs "home"）、缩写、隐喻、行业俚语或跨语言变体——时，召回率会大幅下降。这是稀疏检索（sparse retrieval）的固有局限，也是向量化检索单独使用时仍难以完全覆盖的盲区。

---

## 解决方案

### 方案一：稠密向量检索（Dense Retrieval）替代或补充关键词检索

**核心思路**
将查询和文档都映射为高维稠密向量，通过余弦相似度或内积进行近邻搜索。由于语义相近的词语（包括同义词）在向量空间中距离相近，天然缓解了关键词不匹配问题。

**实现方法**
- 使用预训练双塔模型（Bi-Encoder），如 `sentence-transformers`（`all-mpnet-base-v2`、`e5-large`、`bge-m3`）对查询和文档编码
- 使用 FAISS、Milvus、Qdrant、Weaviate 等向量数据库进行 ANN（近似最近邻）搜索
- 对特定领域可做微调（fine-tuning），提升专业术语的表征质量

**参考来源**
- [Dense Passage Retrieval — EmergentMind](https://www.emergentmind.com/topics/dense-passage-retrieval)
- [RAG vs. Semantic Search — Medium](https://tsaiprabhanj.medium.com/rag-vs-semantic-search-a-deep-dive-for-generative-ai-0ada1e2d7cd0)

### 方案二：混合检索（Hybrid Retrieval）——稀疏 + 稠密融合

**核心思路**
同时运行 BM25（稀疏）与向量检索（稠密），通过 Reciprocal Rank Fusion（RRF）或加权线性融合将两路结果合并。BM25 擅长精确关键词匹配，向量检索擅长语义泛化，二者互补。

**实现方法**
- 主流实现：Elasticsearch/OpenSearch 的 hybrid query、Weaviate BM25+Vector、LangChain `EnsembleRetriever`
- 融合策略：RRF 公式 `score = Σ 1/(k + rank_i)`，k 通常取 60
- 进阶变体：Blended RAG（BM25 + dense vector + sparse vector 三路融合），在 BEIR 基准上优于纯向量与二路混合

**参考来源**
- [Advanced RAG: Hybrid Search and Re-ranking — DEV Community](https://dev.to/kuldeep_paul/advanced-rag-from-naive-retrieval-to-hybrid-search-and-re-ranking-4km3)
- [Hybrid Retrieval-Augmented Generation Systems — Medium](https://medium.com/@adnanmasood/hybrid-retrieval-augmented-generation-systems-for-knowledge-intensive-tasks-10347cbe83ab)
- [Dense vector + Sparse vector + Full text search — Infiniflow](https://infiniflow.org/blog/best-hybrid-search-solution)

### 方案三：查询扩展（Query Expansion）

**核心思路**
在检索前对用户查询进行改写或扩展，自动添加同义词、相关术语、缩写或上下位词，从而增大与索引文档重叠的可能性。

**实现方法**
- **同义词扩展**：基于 WordNet、领域词典或大模型（LLM）生成同义表达
- **多查询生成（Multi-Query）**：用 LLM 对原始查询生成 3-5 个不同角度的改写版本，分别检索后合并结果
- **子查询分解**：将复杂查询拆解为多个子问题，独立检索后聚合
- **LLM 辅助重写**：用 prompt 指导模型将口语化或隐喻性表达转换为更规范的检索词

**参考来源**
- [Query Expansion in Enhancing RAG — Medium](https://medium.com/@sahin.samia/query-expansion-in-enhancing-retrieval-augmented-generation-rag-d41153317383)

### 方案四：假设文档嵌入（HyDE, Hypothetical Document Embeddings）

**核心思路**
HyDE 利用 LLM（如 InstructGPT）为原始查询生成一段"假设性答案文档"，再将该假设文档的向量用于检索，而非直接对查询向量化。由于假设文档与真实文档在语言风格上更接近，能有效跨越查询与文档之间的语言变体鸿沟。

**实现方法**
- 调用 LLM（GPT-4、Mistral 等）生成假设性段落（通常 100-200 词）
- 对假设文档编码为向量，用于 ANN 检索
- 可结合多个假设文档的向量均值提升稳定性
- LangChain、Haystack 均提供开箱即用的 HyDE 组件

**参考来源**

- [Better RAG with HyDE — Zilliz Learn](https://zilliz.com/learn/improve-rag-and-information-retrieval-with-hyde-hypothetical-document-embeddings)

### 方案五：同义词感知的 RAG 系统（Synonym-Aware RAG）

**核心思路**
在索引和检索阶段显式维护一个同义词词典（synonym dictionary）或实体映射表，在查询时自动将用户输入中的词汇映射到标准术语，再结合向量检索获取语义相关内容。

**实现方法**
- 构建领域专属同义词库（可从领域文档中自动抽取或手工维护）
- 在检索前做实体规范化（entity normalization）
- Palo Alto Networks 实践：将同义词感知模块集成到 RAG pipeline 中，显著减少因行业术语变体导致的召回盲点

**参考来源**
- [Bridging the Language Gap: Synonym-Aware RAG at Palo Alto Networks](https://live.paloaltonetworks.com/t5/engineering-blogs/bridging-the-language-gap-our-journey-to-a-synonym-aware-rag/ba-p/1236616)

## 推荐工具 / 框架 / 库

| 工具/库 | 用途 |
|---|---|
| `sentence-transformers` | 稠密编码，支持多种双塔模型 |
| `LangChain` | MultiQueryRetriever、HyDE、EnsembleRetriever 等开箱即用组件 |

| `BGE-M3` | 多语言、多粒度稠密检索模型（特别适合多语言场景） |
