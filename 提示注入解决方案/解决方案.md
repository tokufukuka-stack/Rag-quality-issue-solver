# 提示注入解决方案

## 问题描述（原文）

> 提示注入——恶意文档中嵌入指令操纵模型行为

在 RAG 系统中，提示注入（Prompt Injection）分为两类：
- **直接提示注入**：用户在查询中直接插入恶意指令
- **间接提示注入**：攻击者在被检索的外部文档中预埋指令，当该文档被纳入上下文时，模型将文档中的指令视为合法命令并执行，从而被操控输出攻击者期望的内容、泄露系统提示词或执行未授权操作

RAG 系统尤其脆弱：研究表明，仅向知识库注入少量恶意文档，即可使 LLM 超过 90% 的概率返回攻击者指定的答案。

---

## 主要解决方案

### 方案一：输入/文档预处理与内容净化（Input Sanitization）

**核心思路**

在文档入库（索引阶段）和检索返回（上下文构建阶段）两个环节，对所有外部内容执行严格净化，消除可能携带恶意指令的载体。

**实现方法**

1. 将所有外部文件（PDF、HTML、Word 等）转换为纯文本，剥离 HTML 注释、CSS 隐藏元素、不可见 Unicode 字符（如零宽字符）
2. 去除文件元数据（EXIF、PDF 元信息），因为模型有时会处理这些字段
3. 维护最小化标记白名单，仅保留必要的结构性 Markdown
4. 对 HTML 页面使用专用解析器，只保留可见文本内容
5. Unicode 规范化，防止利用等效字符绕过过滤器

- [Hidden-in-Plain-Text: A Benchmark for Social-Web Indirect Prompt Injection in RAG (Arxiv 2601.10923)](https://arxiv.org/html/2601.10923v1)

### 方案二：指令层级与权限分离（Instruction Hierarchy & Privilege Separation）

**核心思路**

通过架构设计明确区分不同来源指令的可信级别，让模型在权重层面学会优先遵循高权限指令（系统提示 > 开发者 > 用户），并将检索到的文档内容视为"数据"而非"指令"。

**实现方法**

1. **指令层级微调**（OpenAI 提出的 Instruction Hierarchy 框架）：对 LLM 进行指令微调，使其在推理时为不同来源的输入赋予不同权限等级；在所有 Transformer 中间层注入权限级别嵌入，防止指令边界被稀释
2. **角色分离**：使用"规划 Agent + 隔离 Agent"架构，规划 Agent 生成执行计划，隔离 Agent 独立解析不可信数据，防止数据流污染控制流
3. **系统提示强化**：在系统提示中明确限定模型角色，要求模型只响应特定类型的任务，忽略文档中出现的"忘记先前指令"等模式

**参考来源**

- [Securing AI Agents Against Prompt Injection Attacks (Arxiv 2511.15759)](https://arxiv.org/abs/2511.15759)
- [GitHub: tldrsec/prompt-injection-defenses](https://github.com/tldrsec/prompt-injection-defenses)
- [OWASP LLM01:2025 Prompt Injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

### 方案三：运行时检测与输出过滤（Runtime Detection & Output Filtering）

**核心思路**

在检索内容进入 LLM 上下文之前，或在 LLM 输出之后，使用独立的检测模型或规则引擎识别并拦截注入攻击。

**实现方法**

1. **嵌入式异常检测**：对检索到的文档块计算嵌入向量，与正常语料分布进行对比，识别异常的"指令型"文本（如包含大量祈使句、伪系统提示词等）
2. **分类器检测**：训练专用的注入检测分类器，研究表明可实现精度 0.92、召回率 0.89、F1=0.90 的检测效果（1200 个样本评估集）
3. **输出验证**：使用确定性代码对输出格式进行验证；在生成回答时要求模型引用来源并说明推理链，偏离格式的输出触发重新生成或人工审核
4. **AI 网关**：在 RAG 管道前后部署 AI 网关（如 Datadog LLM Observability、Solo.io AI Gateway），对输入和输出进行实时流量分析

**参考来源**

- [Mitigating Indirect Prompt Injection Attacks on LLMs – Solo.io](https://www.solo.io/blog/mitigating-indirect-prompt-injection-attacks-on-llms)
- [Best practices for monitoring LLM prompt injection attacks – Datadog](https://www.datadoghq.com/blog/monitor-llm-prompt-injection-attacks/)
- [Defense Against Prompt Injection Attack by Leveraging... (ACL 2025)](https://aclanthology.org/2025.acl-long.897.pdf)

### 方案四：多层防御体系（Defense-in-Depth）

**核心思路**

单一防御层无法可靠抵御所有攻击，必须将内容净化、权限分离、运行时检测、沙箱隔离、监控审计等多种手段组合使用。

**实现方法**

1. **分层架构**：内容净化 → 权限检查 → 检测过滤 → 沙箱执行 → 输出验证 → 审计日志，每层均可独立拦截部分攻击
2. **沙箱隔离**：将 Agent 的所有工具调用在隔离环境（Docker、VM、云沙箱）中执行，限制网络访问、文件系统权限，最小化成功攻击后的爆炸半径
3. **最小权限原则**：所有参与 LLM Prompt 构建的实体均应使用其所拥有的最低权限级别
4. **持续红队测试**：定期以攻击者视角对系统进行测试，结合人工监督（Human-in-the-Loop）处理高风险操作

研究表明，综合三种机制（内容过滤 + 层级系统提示 + 多阶段响应验证）可将成功攻击率从 73.2% 降低至 8.7%，同时保留 94.3% 的基准任务性能。

### 方案五：访问控制与来源溯源（Access Control & Provenance Tracking）

**核心思路**

在嵌入检索层直接执行访问控制，并对每个检索块打上来源标签，使模型在生成时可明确区分哪些内容来自可信文档、哪些来自未经验证的外部源。

**实现方法**

1. **Embedding 层 ACL**：在向量数据库中为每个 embedding 附加访问控制元数据，检索时验证当前用户/请求的权限，防止越权访问和数据外泄型注入
2. **来源标注**：在将文档块注入 Prompt 时，附加来源路径和可信级别标签（如 `[Source: internal-doc, trust: high]`），引导模型对低信任来源的内容保持怀疑
3. **归因门控回答（Attribution-Gated Answering）**：要求模型只从可归因于检索来源的内容中生成回答，对无法归因的指令拒绝执行
