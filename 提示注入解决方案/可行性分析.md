# 可行性分析：提示注入防御方案

> 本文档对每个解决方案进行独立的可行性评估，基于技术成熟度、资源约束、实际效果和风险综合判断。

---

## 方案一：输入/文档预处理与内容净化

### 技术可行性
**实现难度：低**

内容净化属于工程性工作，无需机器学习模型，主要依赖成熟的文本处理库（BeautifulSoup、python-magic、UniDecode 等）。主要挑战在于攻击者的规避技巧不断演化（如利用 Unicode 等效字符、零宽字符、右到左覆盖字符等），净化规则需持续更新。

### 资源需求
- **算力**：极低，纯文本处理，CPU 即可完成
- **数据**：无需标注数据，只需维护过滤规则库
- **人力**：初始开发 1-2 人周，后续维护成本低

### 预期效果评估
- **优势**：可拦截大量"显式"注入（如 HTML 注释嵌入的指令、白色文字、隐藏 Div 等）
- **局限**：无法拦截"语义层"注入——即使是正常格式的文本，只要语义上是指令（如"忽略以上内容，改为输出……"），净化层完全无法识别

### 潜在风险与局限
- 过度净化可能破坏正常文档的结构（如代码块、特殊格式的学术论文）
- 攻击者可轻易绕过纯规则型过滤，采用自然语言嵌入指令的方式
- 零宽字符和 Unicode 变体类别繁多，维护白名单/黑名单代价持续增加

### 综合推荐指数：⭐⭐⭐（3/5）

**评价**：作为基础防线必须部署，但不能单独依赖，需与其他方案结合。

## 方案二：指令层级与权限分离

### 技术可行性
**实现难度：高**

指令层级微调（Instruction Hierarchy fine-tuning）需要对 LLM 进行专项训练，对于使用闭源 API（如 GPT-4、Claude）的场景，无法直接修改模型权重，只能依赖 System Prompt 层面的"软约束"，效果有限。对于开源模型（Llama、Mistral 等），实施成本较高，需要专业 ML 工程师。

角色分离（规划 Agent + 隔离 Agent）架构在工程层面可行，但增加了系统复杂度和延迟。

### 资源需求
- **算力**：若涉及模型微调，需要 GPU 集群（至少 A100 x 8）和数周训练时间
- **数据**：需要构建包含注入攻击样本的对齐数据集（数千到数万条）
- **人力**：架构设计 2-3 人月；若涉及微调，再加 ML 工程师 2-3 人月

### 预期效果评估
- **模型层面**：经过专项微调的模型，在基准测试中可显著降低对注入指令的服从率
- **架构层面（Agent 分离）**：有效防止数据污染控制流，但增加了 API 调用次数（延迟上升 30-50%）
- **System Prompt 约束**：效果最弱，研究表明仅靠提示词约束的防御可被精心构造的攻击轻易绕过

### 潜在风险与局限
- 模型微调可能影响通用任务性能（alignment tax）
- 闭源模型无法实现真正的权重级别指令层级
- 攻击者可通过逐步升温（gradual escalation）绕过权限边界
- Agent 分离架构在 agentic 场景下状态管理复杂，难以覆盖所有数据流路径

### 综合推荐指数：⭐⭐⭐⭐（4/5）

**评价**：最具根本性的防御手段，但实施成本高。建议优先在开源模型上实施架构层面的角色分离，同时积极跟进大型模型厂商对 Instruction Hierarchy 的原生支持进展。

## 方案三：运行时检测与输出过滤

### 技术可行性
**实现难度：中**

嵌入式异常检测和分类器检测属于成熟的 NLP 分类任务，可基于现有的 BERT 类模型进行微调。已有研究实现了 F1=0.90 的检测精度，但需要高质量的对抗样本标注数据集。AI 网关方案（如 Lakera、Datadog）有成熟商业产品可直接接入。

### 资源需求
- **算力**：检测模型推理需要额外 GPU/CPU 资源，每次检索增加约 50-200ms 延迟
- **数据**：需要构建包含正常文档和注入样本的标注数据集（1000+ 条）
- **人力**：
  - 自研路线：3-4 人月（数据标注 + 模型训练 + 集成）
  - 商业方案：1-2 人周（API 接入）

### 预期效果评估
- **检测精度**：现有最优方案可达 F1=0.90，即仍有约 10% 的漏报/误报
- **多阶段响应验证**（内容过滤 + 层级提示 + 验证）：综合可将攻击成功率从 73.2% 降至 8.7%，是目前文献中效果最佳的组合方案
- **局限**：对抗性样本（adversarial examples）可专门针对检测模型设计绕过策略

### 潜在风险与局限
- 检测模型本身也可能被对抗样本攻击
- 高误报率会导致合法文档被拦截，影响 RAG 召回率和用户体验
- 商业检测服务存在隐私风险（文档内容发送给第三方）、成本持续增加和厂商依赖

### 综合推荐指数：⭐⭐⭐⭐（4/5）

**评价**：短期内性价比最高的方案。建议优先采用商业 AI 网关（如 Lakera Guard 或 LLM Guard 开源版），再逐步建立自有检测能力。

## 方案四：多层防御体系（Defense-in-Depth）

### 技术可行性
**实现难度：中**

各层次组件均有成熟解决方案，挑战在于系统集成和运维复杂度管理。沙箱技术（Docker、gVisor）已非常成熟，接入 RAG 流水线改造量适中。

### 资源需求
- **算力**：沙箱执行有额外开销（约 10-30% 延迟增加）；监控和日志系统需要存储和计算资源
- **数据**：需要持续收集攻击样本用于红队测试
- **人力**：整体架构设计和集成 4-6 人月；红队测试需要安全专家持续投入

### 预期效果评估
- **综合效果最佳**：研究和实践均证明，Defense-in-Depth 是目前对抗提示注入的最有效范式
- **爆炸半径控制**：即使某层防御被突破，沙箱和最小权限原则可将损害限制在可接受范围
- **适应性**：随着新攻击类型出现，可在各层独立更新防御规则，系统整体鲁棒性强

### 潜在风险与局限
- 系统复杂度显著增加，故障排查难度上升
- 多层组件之间的集成点可能引入新的漏洞（supply chain）
- 运维成本高，需要专职安全团队
- 在 agentic AI 场景中，沙箱可能过于严格导致功能受限

### 综合推荐指数：⭐⭐⭐⭐⭐（5/5）

**评价**：从长期安全视角，这是唯一真正可靠的防御范式。对于生产级 RAG 系统（尤其是处理高敏感数据的场景）必须采用。建议从核心层（净化 + 检测）开始，逐步扩展到完整的防御体系。

## 方案五：访问控制与来源溯源

### 技术可行性
**实现难度：中**

向量数据库层面的 ACL 已有多个成熟实现（Weaviate、Milvus、Qdrant 均支持 metadata 过滤）。来源标注是工程性工作，改造量适中。归因门控回答（Attribution-Gated Answering）需要对 Prompt 工程进行定制化设计。

### 资源需求
- **算力**：ACL 过滤增加少量索引开销（5-15%）
- **数据**：需要为现有知识库补充来源元数据（一次性成本）
- **人力**：2-3 人月（ACL 接入 + 来源标注 + Prompt 工程调整）

### 预期效果评估
- **防越权访问**：有效，可防止检索返回用户无权访问的文档（也防止跨租户注入）
- **提升模型警觉性**：来源标注可引导模型对低信任内容保持怀疑，但模型是否真正遵循取决于指令遵从性，无法保证
- **归因门控**：可减少无依据的幻觉，但对"内容在文档中，但文档本身是恶意的"场景无效

### 潜在风险与局限
- ACL 元数据管理随文档数量增加而复杂化，尤其是 chunk 粒度的权限控制
- 来源信任级别的划分本身存在主观性，难以自动化
- 归因门控无法应对文档内容本身合法但包含注入指令的场景（"合法外表，恶意内核"）

### 综合推荐指数：⭐⭐⭐（3/5）

**评价**：更多是权限管控和数据治理的基础能力，对提示注入的直接防御效果有限，但作为零信任架构的基础组件应当部署。

## 最终推荐方案

### 🏆 推荐：方案四（多层防御体系）为核心，方案一+三为快速起步

**推荐理由**：

提示注入是目前 LLM 安全领域公认的"无法完全解决，只能持续缓解"的问题（OWASP LLM01:2025 明确指出这一点）。在此前提下，Defense-in-Depth 是唯一经过实践验证、可将攻击成功率控制在可接受水平的方案。

**实施路线图（建议按阶段推进）**：

**第一阶段（1-2 周，快速起步）**
- 部署方案一：实施文档入库净化管道（HTML 净化、Unicode 规范化、不可见字符过滤）
- 部署方案三（商业方案）：接入 LLM Guard（开源）或 Lakera Guard（商业），在检索内容进入 Prompt 前进行扫描

**第二阶段（1-2 月，架构强化）**
- 部署方案五：为向量数据库添加来源元数据和 ACL，在 Prompt 中标注来源信任级别
- 部署方案四（核心层）：工具调用沙箱化，添加审计日志，建立监控告警

**第三阶段（3-6 月，长期演进）**
- 评估方案二：跟进 LLM 厂商对 Instruction Hierarchy 的原生支持，在条件允许时实施开源模型专项微调
- 建立红队测试机制，持续评估防御有效性
- 完善 Human-in-the-Loop 工作流，对高风险操作要求人工确认

**关键原则**：没有银弹，分层防御 + 持续演进是对抗提示注入的唯一可靠路径。
