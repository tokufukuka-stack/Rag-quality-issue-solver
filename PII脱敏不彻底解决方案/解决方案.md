# PII 在 RAG Chunk 中扩散与脱敏不彻底问题解决方案

## 问题描述（原文）

**PII（个人身份信息）在 chunk 中扩散，脱敏不彻底**

在 RAG 系统中，个人身份信息（PII，如姓名、手机号、身份证号、邮箱、银行卡号等）可能随文档切分（chunking）过程扩散到多个向量块中。若脱敏处理不彻底，这些敏感数据将以两种形式泄露：
1. **原文扩散**：PII 明文随 chunk 存储到向量数据库，检索时被直接暴露给 LLM 或用户；
2. **嵌入编码泄露**：即使不存原文，向量嵌入本身仍隐式编码了 PII 信息，攻击者可通过模型反推获取。

---

## 主要解决方案

### 方案一：预处理阶段 NER + 规则引擎 PII 检测与掩码替换

**方案名称**：Hybrid NER + Rule-based PII Detection & Masking（代表工具：Microsoft Presidio）

**核心思路**：
在文档进入切分和嵌入流程之前，使用命名实体识别（NER）模型与正则表达式规则引擎的混合方案，扫描所有文本，检测并替换或掩码 PII 实体（姓名→`<PERSON>`，手机号→`<PHONE>`，身份证→`<ID_NUM>` 等）。确保进入向量库的 chunk 本身不含明文 PII。

**实现方法**：
1. 使用 Microsoft Presidio（开源）作为检测引擎，结合 spaCy NER 模型识别人名、地点、组织等实体；
2. 正则规则覆盖结构化 PII（信用卡、SSN、邮箱、电话等）；
3. 将检测到的 PII 替换为占位符（如 `<PERSON_1>`），或进行哈希映射（可逆或不可逆）；
4. 将脱敏后文本送入切分和嵌入流程；
5. 原始映射表（PII → 占位符）加密存储于独立安全存储，供授权场景下解码。

### 方案二：可逆匿名化（Reversible Anonymization）+ 检索后还原

**方案名称**：Reversible PII Anonymization with Secure Mapping Table

**核心思路**：
对 PII 执行可逆替换，将假名化后的文本存入向量库；在 RAG 检索结束、生成结果之前，根据用户权限和业务需求决定是否将占位符还原为真实值。将匿名化与去匿名化解耦，在保护隐私的同时保留完整的语义功能。

**实现方法**：
1. 文档预处理时，用 Presidio 或 Faker 生成一致性假名（如将"张三"统一替换为"UserA_1234"），并在安全存储中维护映射表；
2. 所有 chunk 和嵌入均使用假名版本；
3. LlamaIndex 的 `PresidioPIINodePostprocessor` 在检索后对 chunk 中的占位符二次过滤；
4. 响应生成后，若用户具有相应权限，可调用解密接口将假名还原为真实 PII；
5. 无权限用户仅看到脱敏版结果（如"联系人：`<PERSON>`"）。

**参考来源**：
- [Secure LLM Usage With Reversible Data Anonymization | DZone](https://dzone.com/articles/llm-pii-anonymization-guide)
- [Unlocking the Power of RAG with Added Privacy: A Comprehensive Guide | Private-AI](https://www.private-ai.com/en/blog/rag-privacy-guide)

### 方案三：差分隐私扰动（Differential Privacy Perturbation for Private Entities）

**方案名称**：Entity-Level Differential Privacy Perturbation in RAG

**核心思路**：
不同于传统掩码的完全替换，差分隐私扰动对检测到的私有实体（而非整个文本）添加可控噪声，使得攻击者无法从向量嵌入或输出中以高置信度推断原始 PII，同时尽量保留语义完整性。

**实现方法**：
1. 先用 NER 模型定位文本中的私有实体（人名、地址等）；
2. 对每类实体应用专属扰动机制（如对姓名在同类实体集合中随机替换，对数值型 PII 添加拉普拉斯噪声）；
3. 在预检索（pre-retrieval）阶段对向量化前的文本施加扰动；
4. 控制隐私预算（ε）以平衡隐私强度与语义保真度；
5. 结合局部差分隐私（Local DP）技术，使用户查询也得到保护。

**参考来源**：
- [Mitigating privacy risks in RAG via locally private entity perturbation | ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/S0306457325000913)
- [Privacy protection in RAG: A novel method and evaluation framework | ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/S0306457325004467)
- [Mitigating the Privacy Issues in RAG via Pure Synthetic Data | Arxiv](https://arxiv.org/html/2406.14773v2)

### 方案四：合成数据替代（Synthetic Data Replacement for Training/Indexing）

**方案名称**：PII-Free Synthetic Data Generation for RAG Knowledge Base

**核心思路**：
在构建知识库时，对含有 PII 的原始文档，使用生成模型或合成数据工具生成语义等价但不含真实 PII 的版本，将合成版本入库索引，从根本上消除 PII 在 chunk 中出现的可能。

**实现方法**：
1. 使用 Faker、Synthea 等工具为文档中的 PII 字段生成真实感强的虚假替代数据（假名、假地址、假电话）；
2. 对于非结构化文本，使用 LLM 对敏感段落进行"隐私保护式改写"（privacy-preserving paraphrasing）；
3. 合成文档经人工或自动审核后入库；
4. 原始含 PII 文档仅保留在访问受控的安全存储中，不参与 RAG 检索。

**参考来源**：
- [Mitigating the Privacy Issues in RAG via Pure Synthetic Data | Arxiv](https://arxiv.org/html/2406.14773v2)
- [PII Detection and Masking in RAG Pipelines | Analytics Vidhya](https://www.analyticsvidhya.com/blog/2024/03/pii-detection-and-masking-in-rag-pipelines/)

### 方案五：向量数据库访问控制 + 运行时 PII 过滤

**方案名称**：Dual-Layer Access Control + Runtime PII Filtering

**核心思路**：
在架构层面实施"纵深防御"：即使脱敏不完整导致少量 PII 残留在 chunk 中，通过向量数据库的细粒度访问控制和运行时后处理过滤，阻止未授权用户接触到这些内容。

**实现方法**：
1. 在向量数据库（如 Milvus、Weaviate、Elasticsearch）的元数据中标记 chunk 的 PII 敏感级别（高/中/低）；
2. 实施双层策略：Gate Policy（控制哪些检索节点可访问）+ Document-level Policy（基于用户凭证过滤检索内容），默认拒绝（default-deny）；
3. 检索结果返回后，在 LLM 调用前增加运行时 PII 扫描器（如 Presidio Analyzer）作为最后一道防线；
4. 使用 AWS Bedrock Guardrails 或 Azure AI Content Safety 等托管服务对输入/输出进行 PII 过滤；
5. 审计日志记录每次涉及 PII 敏感 chunk 的访问。

**参考来源**：

- [Protect sensitive data in RAG applications with Amazon Bedrock | AWS](https://aws.amazon.com/blogs/machine-learning/protect-sensitive-data-in-rag-applications-with-amazon-bedrock/)
