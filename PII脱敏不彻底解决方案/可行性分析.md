# PII 脱敏不彻底问题——可行性分析报告

> 分析日期：2026-04-04
> 分析对象：针对"PII 在 RAG Chunk 中扩散与脱敏不彻底"问题的五种解决方案

---

## 方案一：Hybrid NER + 规则引擎 PII 检测与掩码替换（Microsoft Presidio）

### 技术可行性
**实现难度：低**

Presidio 是成熟的开源项目，与 LlamaIndex、LangChain 均有原生集成。API 调用简单，spaCy 模型加载和规则配置文档完善。对于英文场景落地极为成熟；中文场景需额外接入中文 NER 模型（如 HanLP、BERT-CRF），需一定适配工作，整体仍可控。

### 资源需求
- **算力**：CPU 级别可运行（spaCy 小模型），如需高精度中文 NER 则需 GPU；
- **数据**：无需额外训练数据（使用预训练模型），如需微调则需标注语料；
- **人力**：1~2 名工程师，1~2 周可完成基础集成；中文定制化需额外 1~2 周。

### 预期效果评估
根据 Nature 论文实测，混合方案在金融文档上达到 F1=91.1%、准确率=89.4%，商业场景优化后可超过 93%。对结构化 PII（电话、邮箱、身份证）几乎无漏检；非结构化 PII（隐晦表达的姓名）仍有约 10% 漏检率。整体可大幅降低 PII 扩散风险。

### 潜在风险与局限
1. **漏检率不为零**：NER 模型对隐晦、变体 PII（如"老王"、缩写姓名）识别能力有限；
2. **过度脱敏**：误将非 PII 关键词识别为 PII 并掩码，影响 RAG 检索质量；
3. **中文适配成本**：Presidio 原生针对英文，中文需要额外模型和规则维护；
4. **静态规则滞后**：新型 PII（如新格式身份证、企业统一社会信用代码）需持续更新规则。

### 综合推荐指数
⭐⭐⭐⭐⭐（5/5）

**理由**：成熟度最高、落地成本最低、社区支持最好，是 RAG 场景 PII 保护的首选起点方案。建议作为所有 RAG 系统的标准基础配置，先解决"有无"问题，再用其他方案优化"好坏"问题。

---

## 方案二：可逆匿名化（Reversible Anonymization）+ 检索后权限还原

### 技术可行性
**实现难度：中**

技术链路清晰（Presidio 匿名化 + 映射表存储 + 检索后还原），但需要额外设计安全的映射表存储方案（如加密 KV 存储、HSM），以及基于用户身份的权限判断逻辑。需与现有 IAM 系统集成，存在一定工程复杂度。

### 资源需求
- **算力**：检测阶段与方案一相同（CPU 可行）；还原阶段仅为查表操作，极轻量；
- **数据**：需要安全存储系统（如 Vault、KMS）存储映射表；
- **人力**：1~3 名工程师，2~4 周完成，含安全存储集成。

### 预期效果评估
可做到"对 LLM 透明脱敏 + 对授权用户无损还原"。在有明确用户权限分级的场景（如医疗、金融）效果优秀，能在隐私保护和业务可用性之间实现较好平衡。检索质量与方案一相当（因嵌入阶段同样使用脱敏文本）。

### 潜在风险与局限
1. **映射表安全是核心风险**：若映射表泄露，等同于 PII 全部泄露，需对映射表实施最高级别安全保护；
2. **一致性挑战**：同一 PII 在不同文档中需确保映射一致（如"张三"始终→"UserA_1234"），跨文档追踪复杂；
3. **还原逻辑的业务侵入性**：每个下游应用都需感知"是否还原"的判断逻辑；
4. **GDPR"被遗忘权"实现仍需额外处理**：删除映射条目不等于删除向量库中的嵌入。

### 综合推荐指数
⭐⭐⭐⭐（4/5）

**理由**：在有访问控制需求的场景非常合适，是对方案一的重要补充。建议在方案一基础上，对涉及多级权限的业务场景叠加此方案。不建议单独使用，因其安全强度高度依赖映射表的保护质量。

## 方案三：差分隐私扰动（Entity-Level Differential Privacy）

### 技术可行性
**实现难度：高**

差分隐私在 NLP 文本上的应用仍处于学术前沿，工程实现缺乏成熟的开箱即用框架。需深度理解隐私预算（ε）与语义保真度之间的 trade-off，实体级扰动机制的设计非常依赖领域知识。对于中文 PII 的扰动机制研究更为稀缺。

### 资源需求
- **算力**：扰动计算本身轻量，但调优需要大量实验；
- **数据**：需要同类实体词表（用于替换扰动），以及评估语义保真度的标注集；
- **人力**：需要具备差分隐私理论背景的研究工程师，实施周期 1~3 个月。

### 预期效果评估
理论上可同时防护明文 PII 和嵌入编码隐式泄露，是最为彻底的隐私保护方案。但实际效果高度依赖 ε 值的选取：ε 过小导致语义严重失真（RAG 检索质量下降），ε 过大则隐私保护形同虚设。目前学术论文中的实测结果在特定数据集上表现良好，但泛化能力尚需验证。

### 潜在风险与局限
1. **语义保真度损耗难以量化**：扰动后的文本语义是否足以支撑高质量 RAG 检索，缺乏通用评估标准；
2. **工程落地门槛极高**：学术方案与生产系统之间存在显著鸿沟；
3. **领域适配困难**：医疗、法律等专业领域的实体词表维护成本很高；
4. **合规认可度存疑**：差分隐私是否满足 GDPR/《个人信息保护法》要求尚无定论。

### 综合推荐指数
⭐⭐⭐（3/5）

**理由**：学术价值高，适合作为研究探索方向或高安全等级系统的长期演进路径。短期内不适合在生产系统中单独采用，建议在方案一/二落地后，有充足研发资源时作为补充研究项目推进。

## 方案四：合成数据替代（Synthetic Data Replacement）

### 技术可行性
**实现难度：中**

Faker 等工具生成结构化 PII 替代（姓名、地址等）的技术成熟；但对于非结构化长文本的"隐私保护式改写"，需要借助 LLM，改写质量高度依赖 prompt 设计，且需要人工审核以确保语义等价性。

### 资源需求
- **算力**：LLM 改写需要 GPU 或 API 调用成本（按文档量线性增长）；
- **数据**：原始文档和合成结果均需存储；人工审核需要具备业务背景的标注人员；
- **人力**：2~4 名工程师 + 业务审核人员，实施周期 1~2 个月。

### 预期效果评估
从根本上消除知识库中的真实 PII，是最为彻底的"源头治理"方案。对 RAG 检索质量影响取决于合成质量——高质量合成数据可做到零质量损耗，低质量合成（语义扭曲）则严重影响检索效果。

### 潜在风险与局限
1. **合成质量难以保证**：LLM 改写可能引入语义偏差，影响知识库知识的准确性；
2. **规模化成本高**：大规模文档库需要大量 LLM 调用，API 成本可观；
3. **不适用于动态文档**：频繁更新的文档库无法实时执行合成替换流程；
4. **原始文档管理复杂**：需维护原始含 PII 文档的安全存储，增加运维负担；
5. **合成数据合规性**：部分监管场景要求原始数据可追溯，合成替换可能引发合规争议。

### 综合推荐指数
⭐⭐⭐（3/5）

**理由**：适合静态知识库、高安全等级场景（如医疗档案）的一次性处理。不适合作为通用 RAG 系统的标准方案，建议仅在对 PII 零容忍的特殊场景中配合方案一使用。

---

## 方案五：向量数据库访问控制 + 运行时 PII 过滤

### 技术可行性
**实现难度：中**

主流向量数据库（Milvus、Weaviate、Elasticsearch）均支持元数据过滤，主要工作在于元数据标记体系设计和 IAM 集成。运行时 Presidio 扫描器的集成较为简单，AWS Bedrock Guardrails 等托管服务则几乎开箱即用。

### 资源需求
- **算力**：元数据过滤近乎零额外算力；运行时扫描增加约 50~200ms 延迟；
- **数据**：需维护文档/chunk 的 PII 敏感级别元数据；
- **人力**：1~2 名工程师，2~3 周完成集成。

### 预期效果评估
本方案是"纵深防御"策略的最后一道防线，不能从根本上消除 PII（chunk 中的 PII 依然存在），但可有效控制 PII 的暴露范围。与方案一/二配合使用时，可将整体 PII 泄露风险降至极低水平。单独使用时，安全性取决于访问控制策略的严密性，存在被绕过的风险。

### 潜在风险与局限
1. **不能消除 PII，只能控制暴露**：chunk 原文仍含 PII，若数据库被直接访问则全部泄露；
2. **元数据标记不完整**：若 chunk 的 PII 敏感级别标记不准确，访问控制形同虚设；
3. **运行时延迟增加**：每次检索后增加扫描步骤，对延迟敏感场景（如实时对话）有影响；
4. **策略维护成本**：随业务发展，访问策略的维护和审计需要持续投入。

### 综合推荐指数
⭐⭐⭐⭐（4/5）

**理由**：作为防御层次的重要组成，建议作为系统架构标配。与方案一、方案二组合时可形成完整的多层防护体系，综合推荐指数提升至 5/5。

## 最终推荐方案及理由

### 推荐组合：方案一 + 方案二 + 方案五（三层纵深防御）

**具体实施路径：**

1. **第一层（源头治理）**：在文档入库预处理阶段，部署 Presidio + 中文 NER 模型，对所有 chunk 进行 PII 检测并替换为一致性占位符（可逆版本）。这一步覆盖 ~90% 的 PII，是最关键的环节。

2. **第二层（权限分级）**：建立加密映射表，对具有合法访问需求的用户（如客服、数据分析师），在授权后提供 PII 还原接口；其他用户只接触脱敏版本。

3. **第三层（运行时防护）**：在向量数据库中为所有 chunk 标记 PII 敏感级别，实施元数据过滤和访问控制；同时在检索结果进入 LLM 前增加 Presidio 实时扫描作为最后防线。

**推荐理由：**
- **可行性最高**：三层方案均有成熟工具支持，工程团队 1 个月内可完成基础版本落地；
- **覆盖面最广**：从数据入库到检索输出的全链路均有 PII 防护，无单点失效；
- **渐进式部署**：三层方案可独立部署，优先上线方案一，再逐步添加方案二和方案五；
- **合规性强**：该组合方案符合《个人信息保护法》、GDPR 等主流数据保护法规要求；
- **对 RAG 质量影响最小**：脱敏操作在预处理阶段完成，不影响运行时检索和生成性能。
