# 可行性分析：文档级 ACL 映射到 Chunk 级别

> 本分析基于对各方案的独立评估，综合考量技术可行性、资源需求、预期效果、潜在风险等维度。

---

## 方案一：Chunk 元数据继承 + 向量数据库过滤

### 技术可行性
**实现难度：低**

元数据继承是目前最常见的实现路径，各大向量数据库（Pinecone、Qdrant、Weaviate、Chroma）均原生支持元数据字段和过滤查询。切分管道添加元数据传递逻辑改动小，无需引入新的外部依赖。

### 资源需求
- **算力：** 低，仅增加元数据字段存储和过滤计算
- **数据：** 需要维护文档-权限的映射表；权限变更时需批量更新 chunk 元数据（存在写放大问题）
- **人力：** 中低，主要工作在切分管道改造和权限同步逻辑

### 预期效果
- 对于简单的 RBAC 或少量固定角色的场景效果良好
- 检索性能基本不受影响（现代向量数据库的元数据过滤通常经过优化）
- 能有效阻止未授权用户检索到敏感 chunk

### 潜在风险与局限
1. **权限同步延迟**：权限变更后，向量数据库中的 chunk 元数据若未及时同步，存在短暂的权限漏洞窗口
2. **元数据爆炸**：若用户属于大量群组，`allowed_groups` 字段可能存储大量数据，过滤条件也可能超出某些数据库的查询大小限制
3. **权限传播错误**：切分边界跨越文档时，权限继承逻辑若设计不当可能导致 chunk 遗漏权限或携带错误权限
4. **动态权限支持弱**：难以支持基于时间、上下文的动态权限决策

### 综合推荐指数
⭐⭐⭐⭐（4/5）

适合大多数中小规模、权限结构相对简单的 RAG 系统，是最低成本、最易落地的方案。

---

## 方案二：数据库行级安全（RLS）

### 技术可行性
**实现难度：中**

依赖数据库引擎（主要是 PostgreSQL）的 RLS 特性。需要将 RAG 系统的向量存储迁移至 pgvector，并在应用层正确设置会话变量。若已有系统使用专用向量数据库（如 Pinecone），迁移成本较高。

### 资源需求
- **算力：** 中，PostgreSQL 的向量查询性能不如专用向量数据库，大规模场景下需要额外调优
- **数据：** 需要在同一数据库中管理向量索引和权限数据
- **人力：** 中，需要 DBA/数据库工程师介入，以及应用层的会话管理改造

### 预期效果
- 权限控制在数据库层强制执行，几乎不可能被应用层逻辑绕过，安全性高
- 权限变更通过 SQL 即时生效，无同步延迟
- 对于多租户隔离（每个用户/租户只看自己数据）场景，效果极佳

### 潜在风险与局限
1. **扩展性限制**：PostgreSQL + pgvector 在向量检索性能上不如 Pinecone、Qdrant 等专用方案，亿级文档场景下存在性能瓶颈
2. **技术栈绑定**：深度依赖 PostgreSQL，更换向量数据库时需要重新设计权限层
3. **复杂关系型权限难以表达**：RLS 策略基于当前会话变量，对于多级继承的复杂关系型权限（如部门层级）表达能力有限
4. **连接池管理复杂**：每个请求需要正确设置会话级用户 ID，连接池配置不当可能导致权限泄漏

### 综合推荐指数
⭐⭐⭐（3/5）

适合已使用 PostgreSQL/Supabase 技术栈、数据量在千万级以下、需要强安全隔离的多租户场景。不适合大规模高并发或已使用专用向量数据库的系统。

## 方案三：关系型授权模型（ReBAC）+ 专用授权引擎

### 技术可行性
**实现难度：高**

引入 OpenFGA、SpiceDB 等授权引擎需要额外部署和维护，需要为 RAG 系统定制关系模型（Relationship Tuples），并在检索管道中集成授权引擎调用。工程复杂度显著提升，但提供了最强的权限表达能力。

### 资源需求
- **算力：** 中高，授权引擎需要独立部署，额外增加网络调用延迟（通常 < 10ms，但需优化）
- **数据：** 需要维护授权引擎中的关系数据（Tuples），与原始权限系统保持同步
- **人力：** 高，需要授权系统设计能力，对关系模型（ReBAC）有较好理解

### 预期效果
- 能够处理任意复杂的权限关系，包括组织层级、项目成员、临时授权等
- 权限变更仅需在授权引擎中更新，无需修改向量数据库，一致性好
- 可扩展到企业级复杂场景（数百万用户-文档关系）
- OpenFGA/SpiceDB 经过大规模生产验证（OpenAI、Netflix 使用 SpiceDB）

### 潜在风险与局限
1. **引入新的服务依赖**：授权引擎是新的关键路径组件，需要保证其高可用性；服务故障将导致 RAG 完全不可用
2. **ListObjects API 性能**：对于拥有大量可访问文档的用户（如管理员），`ListObjects` 返回的 ID 集合可能非常大，作为 metadata filter 可能导致查询性能下降
3. **数据同步复杂性**：源系统的权限变更需要及时同步到授权引擎，同步管道的设计和监控增加运维负担
4. **学习曲线陡**：ReBAC 建模需要专门的领域知识，初期建模错误难以发现

### 综合推荐指数
⭐⭐⭐⭐⭐（5/5）

对于权限关系复杂、安全合规要求高的企业级 RAG 系统，这是理论上最完整的解决方案。适合有一定工程能力、权限需求持续演进的团队。

## 方案四：属性访问控制（ABAC）+ 动态元数据标签

### 技术可行性
**实现难度：中**

ABAC 比纯元数据继承方案更灵活，但仍依赖向量数据库的元数据过滤能力。核心复杂度在于策略设计和用户属性的实时获取，无需额外的外部服务依赖。

### 资源需求
- **算力：** 低，策略计算在应用层完成
- **数据：** 需要维护用户属性系统（Identity Provider 或 HR 系统集成）
- **人力：** 中，需要设计属性模型和策略规则；策略维护随业务演进需要持续投入

### 预期效果
- 比纯 RBAC 更灵活，能基于动态属性（如用户所在项目、当前任务）调整访问权限
- 权限变更可通过修改用户属性或策略实现，无需批量更新 chunk
- 适合基于属性自然划分的权限边界（如按部门、按密级）

### 潜在风险与局限
1. **策略设计复杂**：ABAC 策略容易出现意外的权限组合，测试覆盖难度大
2. **属性爆炸问题**：若 chunk 需要携带大量属性字段，元数据管理复杂度增加
3. **动态上下文支持有限**：无法支持"仅限会议期间临时授权"等高度动态的权限场景
4. **策略漂移风险**：随业务演进，策略规则逐渐累积可能导致逻辑冲突

### 综合推荐指数
⭐⭐⭐⭐（4/5）

适合以属性为主要权限划分维度的企业场景（如按部门/密级/地区划分），实现成本低于 ReBAC，但灵活性略低。可作为方案一的升级版本。

## 方案五：云原生托管方案（AWS Bedrock 等）

### 技术可行性
**实现难度：低**

充分利用云厂商提供的托管能力，通过 IAM 策略和独立 Knowledge Base 实现权限隔离，无需自研权限逻辑。

### 资源需求
- **算力：** 由云厂商托管，按使用量计费
- **数据：** 需要按权限级别预先分类存储文档，维护多个独立的向量存储
- **人力：** 低（运维）；但初期架构设计需要对云服务有深度了解

### 预期效果
- 快速实现基础的权限隔离，安全合规由云厂商负责
- 与云生态（S3、IAM、CloudTrail）深度集成，审计追踪容易
- 适合权限结构简单（按团队/部门隔离）的场景

### 潜在风险与局限
1. **权限粒度粗**：本质上是"一个 Knowledge Base = 一个权限集合"，无法支持 chunk 级别的细粒度权限差异
2. **供应商锁定**：深度依赖 AWS，迁移至其他平台成本高
3. **灵活性差**：动态权限变更（如按项目成员动态调整）支持有限
4. **成本高**：多个独立 Knowledge Base 存储同一文档的不同版本会显著增加存储和嵌入成本

### 综合推荐指数
⭐⭐（2/5）

仅适合已深度使用 AWS 生态、权限结构极为简单（如简单的团队级别隔离）的场景。不推荐作为长期核心架构。

## 最终推荐方案及理由

### 推荐策略：分阶段实施，最终过渡到方案三

**第一阶段（快速落地）：方案一 + 方案四 的组合**

对于大多数团队，最务实的起点是：
1. 在文档切分管道中为每个 chunk 继承父文档的权限元数据（方案一基础）
2. 将简单的角色权限升级为基于属性的过滤（方案四思路），支持按部门/密级过滤
3. 建立权限同步管道，确保权限变更能在秒级内批量更新 chunk 元数据

**成本：** 低，2-4 周可上线；**覆盖场景：** 80% 的企业权限需求

**第二阶段（成熟阶段）：引入方案三（ReBAC 授权引擎）**

当业务权限关系变得复杂（如跨项目文档共享、临时授权、细粒度的 chunk 级别差异权限），引入 OpenFGA 或 SpiceDB：
1. 对每个 chunk 在授权引擎中注册为独立资源，关联父文档权限策略
2. 检索前调用 `ListObjects` 获取可访问 chunk ID，作为精准过滤条件
3. 权限变更仅在授权引擎中更新，与向量数据库完全解耦

**成本：** 中高；**覆盖场景：** 99% 的权限需求，支持未来任意权限模型演进

**推荐理由：**
- 方案一/四的组合投入产出比最高，足以解决大多数当前业务场景
- 方案三是长期正确方向，但引入时机应在业务真正需要复杂权限时，避免过度工程
- 方案二（RLS）可作为使用 PostgreSQL 的小团队的过渡方案
- 避免方案五的供应商锁定风险，权限逻辑应掌握在自己手中
