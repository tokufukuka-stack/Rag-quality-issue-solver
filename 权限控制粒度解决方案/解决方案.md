# 权限控制粒度不足：文档级 ACL 难以映射到 Chunk 级别

## 问题描述（原文）

> 权限控制粒度不足——文档级 ACL 难以映射到 chunk 级别

在 RAG 系统中，原始文档通常附带文档级别的访问控制列表（ACL），例如"只有财务部门可以查看本报告"。然而，当文档被切分为多个 chunk 并存入向量数据库后，如何将文档级别的权限精确传递并在检索时按 chunk 粒度执行，成为一个复杂的工程与安全问题。

**典型挑战：**
- 文档切分后，每个 chunk 需独立携带权限元数据
- 权限变更（如撤销某用户访问权）需要同步更新所有相关 chunk
- 复杂的关系型权限（如基于部门/角色/项目的访问控制）难以简单地展平为 chunk 元数据标签
- 用户属于大量群组时，元数据过滤查询可能超出系统限制

---

## 主要解决方案

### 方案一：Chunk 元数据继承 + 向量数据库过滤（Metadata Inheritance + Filter）

**核心思路：**
在文档切分阶段，将原始文档的 ACL/权限信息作为元数据字段附加到每个 chunk 上（继承传播）。在检索时，先查询授权系统获取当前用户可访问的文档 ID 列表，再以此作为过滤条件执行向量相似度搜索。

**实现方法：**
1. 切分文档时，为每个 chunk 附加 `source_doc_id`、`allowed_roles`、`allowed_users` 等字段
2. 用户发起查询时，先调用授权服务返回其可访问的 `doc_id` 集合
3. 向量检索时添加 metadata filter：`{ "source_doc_id": { "$in": accessible_doc_ids } }`
4. 权限变更时，通过 `source_doc_id` 批量更新或删除对应 chunk

**适用数据库：** Pinecone、Weaviate、Qdrant、Chroma、pgvector 等

**参考来源：**
- [Pinecone: RAG with Access Control](https://www.pinecone.io/learn/rag-access-control/)
- [Zilliz Blog: Chunk-Level Data Partitioning](https://zilliz.com/blog/beyond-rag-partitions-per-user-per-chunk-access-policy)
- [Paragon: Building a Permissions System For RAG](https://www.useparagon.com/learn/ai-knowledge-chatbot-with-permissions-chapter-2/)

### 方案二：数据库行级安全（Row-Level Security / RLS）

**核心思路：**
利用支持 RLS 的数据库（如 PostgreSQL + pgvector）将访问控制逻辑下推到数据库层。通过数据库原生的行级安全策略，确保用户在执行向量相似度查询时只能"看到"其有权访问的行（chunk）。

**实现方法：**
1. 在 pgvector 表中为每个 chunk 记录 `owner_id` 或 `tenant_id`
2. 启用 PostgreSQL RLS：
   ```sql
   ALTER TABLE document_chunks ENABLE ROW LEVEL SECURITY;
   CREATE POLICY user_isolation ON document_chunks
     USING (tenant_id = current_setting('app.current_user_id'));
   ```
3. 每个数据库连接/会话设置当前用户 ID，查询自动被过滤
4. 权限修改直接通过 SQL 更新，无需应用层额外逻辑

**适用场景：** 基于 Supabase/PostgreSQL 的 RAG 系统，多租户 SaaS 场景

**参考来源：**
- [Supabase Docs: RAG with Permissions](https://supabase.com/docs/guides/ai/rag-with-permissions)
- [Medium: Implementing Row Level Security in Vector DBs](https://medium.com/@michael.hannecke/implementing-row-level-security-in-vector-dbs-for-rag-applications-fdbccb63d464)

### 方案三：关系型授权模型（ReBAC）与专用授权引擎

**核心思路：**
使用专门的细粒度授权系统（如 OpenFGA、SpiceDB、Cerbos），基于"关系图"而非静态 ACL 列表来建模用户-文档-权限的关系。授权引擎在检索前实时计算用户对每个资源的访问权限，避免将复杂权限关系展平为元数据的局限性。

**实现方法：**
1. 在授权引擎中定义关系模型：`user -> (member of) -> group -> (can_access) -> document`
2. 文档切分时，在授权引擎中为每个 chunk 创建对应资源对象，关联父文档的权限策略
3. 检索前调用授权引擎的 `ListObjects` API 获取用户可访问的 chunk/document ID 集合
4. 或在检索后对返回的 chunk 逐一调用 `Check` API 进行权限校验（后过滤）
5. 权限变更仅在授权引擎中更新，无需修改向量数据库

**代表工具：**
- [OpenFGA](https://openfga.dev/) - 开源，由 Okta 主导，灵感来自 Google Zanzibar
- [SpiceDB](https://authzed.com/) - 开源，Google Zanzibar 的高性能实现，被 OpenAI、Netflix 使用
- [Cerbos](https://cerbos.dev/) - 开源，支持 ABAC/RBAC

**参考来源：**
- [Medium: Fine-Grained Authorization in RAG Systems with OpenFGA](https://medium.com/@sandeepblue15/fine-grained-authorization-in-rag-systems-with-openfga-a-technical-deep-dive-52b5024bf99d)
- [AuthZed Blog: Fine Grained Authorization using SpiceDB for RAG](https://authzed.com/blog/fine-grained-authorization-using-spicedb-for-retrieval-augmented-generation-rag)
- [Descope: Adding Performant ReBAC to RAG Pipelines at Scale](https://www.descope.com/blog/post/rebac-rag)
- [Cerbos Blog: Authorization for RAG Applications](https://www.cerbos.dev/blog/authorization-for-rag-applications-langchain-chromadb-cerbos)

### 方案四：属性访问控制（ABAC）+ 动态元数据标签

**核心思路：**
基于 ABAC 模型，将权限判断依赖于用户属性（部门、职级、地区）和资源属性（文档类别、密级、所属项目）的组合，而非静态的用户-文档绑定。在 chunk 元数据中存储资源属性，检索时根据用户属性动态构建过滤规则。

**实现方法：**
1. 为 chunk 元数据设计属性字段：`{ "department": "finance", "classification": "confidential", "project": "Q4_budget" }`
2. 定义 ABAC 策略：`if user.department == chunk.department AND user.clearance >= chunk.classification THEN allow`
3. 检索时根据当前用户属性动态生成 metadata filter 条件
4. 权限变更通过修改用户属性或策略规则实现，无需逐一更新 chunk

**适用场景：** 大型企业环境、复杂组织架构、跨部门文档共享

**参考来源：**
- [WorkOS Blog: Securing RAG with Fine-Grained Authorization](https://workos.com/blog/how-to-secure-rag-applications-with-fine-grained-authorization-tutorial-with-code)
- [Lasso Security: RAG Access, Permissions & Context](https://www.lasso.security/blog/riding-the-rag-trail-access-permissions-and-context)

### 方案五：AWS Bedrock 托管访问控制 + 数据隔离

**核心思路：**
使用云原生解决方案（如 AWS Bedrock Knowledge Bases），利用平台内置的 IAM 策略、数据源隔离和加密机制，将权限控制委托给云平台管理，避免自行实现复杂的 chunk 级 ACL 传播逻辑。

**实现方法：**
1. 为不同权限等级的文档创建独立的 Knowledge Base 或 S3 存储桶
2. 通过 AWS IAM 策略控制不同用户/角色对不同 Knowledge Base 的访问
3. 利用 Bedrock 的 Guardrails 功能在生成阶段拦截敏感内容
4. 跨账户隔离实现租户级别的完全数据分离

**适用场景：** 已使用 AWS 生态的企业，愿意以牺牲灵活性换取托管安全

**参考来源：**
- [AWS Blog: Protect sensitive data in RAG with Amazon Bedrock](https://aws.amazon.com/blogs/machine-learning/protect-sensitive-data-in-rag-applications-with-amazon-bedrock/)
- [Databricks: ACL and Metadata Filtering with Mosaic AI Vector Search](https://community.databricks.com/t5/technical-blog/mastering-rag-chatbot-security-acl-and-metadata-filtering-with/ba-p/101946)
