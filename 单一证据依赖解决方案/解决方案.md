# 过度依赖单一检索片段，忽视其他相关证据

## 问题描述（原文）

> 过度依赖单一检索片段，忽视其他相关证据

在传统 RAG 管道中，系统通常仅基于向量相似度取 Top-1 或少量 Top-K 结果作为上下文，导致模型在生成答案时只参考了最接近查询的一个片段。当问题需要综合多个文档片段的信息时，这种单一证据依赖模式会造成答案不完整、推理断链或关键信息遗漏。

---

## 主要解决方案

### 方案一：RAG-Fusion（查询扩展 + 互惠排名融合）

**核心思路**
利用 LLM 将原始查询改写为多个语义互补的子查询（通常 3-5 个），分别检索后使用 Reciprocal Rank Fusion（RRF）算法将各列表融合为统一排名，从而显著扩大证据覆盖面。

**实现方法**
1. 用 LLM 对原查询生成 N 条变体查询（如 N=4）
2. 对每条变体查询独立检索向量数据库，得到 N 组 Top-K 结果
3. 使用 RRF 公式 `score(d) = Σ 1/(k + rank_i(d))` 合并排名（k=60 为常用默认值）
4. 取融合后排名靠前的文档作为最终上下文输入 LLM

**预期效果**
相比 vanilla RAG 答案准确率提升 8-10%，内容覆盖全面性提升 30-40%

**参考来源**
- [RAG-Fusion: a New Take on Retrieval-Augmented Generation (arXiv:2402.03367)](https://arxiv.org/abs/2402.03367)
- [Advanced RAG — Understanding Reciprocal Rank Fusion in Hybrid Search](https://glaforge.dev/posts/2026/02/10/advanced-rag-understanding-reciprocal-rank-fusion-in-hybrid-search/)
- [RAG-Fusion Industry Deployment (arXiv:2603.02153)](https://arxiv.org/html/2603.02153)

### 方案二：混合检索（Hybrid Retrieval）

**核心思路**
将密集向量检索（Dense）、稀疏关键词检索（Sparse/BM25）以及知识图谱遍历等多种检索后端并联使用，通过 Rank Fusion 或加权求和聚合多路证据，弥补单一检索方式的覆盖盲区。

**实现方法**
1. 构建多路检索器：dense embedding（如 text-embedding-ada-002）+ BM25 + 可选 KG
2. 各检索器分别返回 Top-100 候选文档
3. 使用 RRF 或加权分数融合为统一候选池（Top-100 → Top-20）
4. 引入 Cross-Encoder 精排（如 `cross-encoder/ms-marco-MiniLM-L-6-v2`），选取最终 Top-5~10

**预期效果**
混合管道相比单一后端在召回率、精确率及多跳推理支持方面均有系统性提升

**参考来源**
- [Hybrid Retrieval-Augmented Generation (RAG) - EmergentMind](https://www.emergentmind.com/topics/hybrid-retrieval-augmented-generation-rag)
- [Building Contextual RAG Systems with Hybrid Search and Reranking](https://www.analyticsvidhya.com/blog/2024/12/contextual-rag-systems-with-hybrid-search-and-reranking/)
- [Neural-Based Rank Fusion for Multi-Source Retrieval in LLMs](https://www.rohan-paul.com/p/neural-based-rank-fusion-for-multi)

### 方案三：Agentic RAG（智能体驱动的迭代检索）

**核心思路**
将检索过程包装在 Agent 循环中，加入路由器（Router）、检索器（Retriever）、评分器（Grader）、生成器（Generator）和幻觉检测器（Hallucination Checker）五个组件。对于复杂问题，Agent 可以分解子问题、迭代补充检索，直到获取足够证据。

**实现方法**
1. 使用 LangGraph / LlamaIndex Agents 构建 ReAct 风格的检索 Agent
2. Agent 判断当前已有证据是否充足，若不足则生成新子查询继续检索
3. 多跳场景下，将前一步检索结果作为上下文扩展后续查询
4. Grader 组件过滤低相关片段，保留高质量证据集合后汇总给 LLM

**预期效果**
在复杂基准测试上准确率可达 78%，相比传统 retrieve-then-generate 管道（34%）大幅提升

**参考来源**
- [Agentic RAG Explained: How Agent-Controlled Retrieval Beats Fixed Pipelines](https://particula.tech/blog/agentic-rag-agent-controlled-retrieval)
- [Synergizing RAG and Reasoning: A Systematic Review (arXiv)](https://arxiv.org/html/2504.15909v1)

### 方案四：多尺度分块 + 证据聚合（Multi-Scale Chunking）

**核心思路**
以不同粒度（如 100/200/500 tokens）对同一语料库分别构建索引，查询时在多个粒度上并行检索，再通过 RRF 融合各粒度的结果，兼顾细节信息与上下文完整性。

**实现方法**
1. 在索引阶段以 3-4 种不同 chunk size 构建多套向量索引
2. 查询时对每套索引独立检索，得到多组候选
3. 使用 RRF 或 Maximal Marginal Relevance（MMR）多样性策略合并结果
4. 将合并后的多尺度片段拼接为最终 context

**预期效果**
在 RAG 基准测试中相比固定 chunk size 方案提升 1-37%（视数据集而定）

**参考来源**
- [Chunk size is query-dependent: a simple multi-scale approach to RAG retrieval (AI21)](https://www.ai21.com/blog/query-dependent-chunking/)
- [Finding the Best Chunking Strategy for Accurate AI Responses (NVIDIA)](https://developer.nvidia.com/blog/finding-the-best-chunking-strategy-for-accurate-ai-responses/)

### 方案五：知识图谱辅助的多跳检索（KG-enhanced RAG）

**核心思路**
将文本 chunk 作为图节点，通过 LLM 生成的伪查询建立节点间的逻辑边，形成段落图（Passage Graph）。检索时采用"检索-推理-剪枝"机制，从种子节点出发沿逻辑边多跳探索，汇聚跨文档的分散证据。

**实现方法**
1. 离线阶段：用 LLM 为每对相邻 chunk 生成连接逻辑关系，构建 Passage Graph
2. 检索阶段（HopRAG 方式）：先用语义/关键词找到种子节点，再沿图边扩展 1-2 跳
3. 可与 Neo4j 等图数据库结合实现高效多跳遍历
4. 将图路径上的所有节点内容聚合后作为 context

**预期效果**
有效解决需要跨文档链式推理的问题，在多跳 QA 基准（如 HotpotQA、MuSiQue）上表现显著优于单轮检索

**参考来源**
- [HopRAG: Multi-Hop Reasoning for Logic-Aware RAG (arXiv:2502.12442)](https://arxiv.org/abs/2502.12442)
- [Agentic RAG with Knowledge Graphs for Complex Multi-Hop Reasoning (arXiv)](https://arxiv.org/html/2507.16507v1)
- [How to Improve Multi-Hop Reasoning With Knowledge Graphs and LLMs (Neo4j)](https://neo4j.com/blog/genai/knowledge-graph-llm-multi-hop-reasoning/)
