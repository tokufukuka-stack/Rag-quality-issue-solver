# 问题：Chunk 切分破坏上下文，句子或段落被不合理截断

## 问题描述（原文）

> Chunk 切分破坏上下文，句子或段落被不合理截断

在 RAG 系统中，文档被分割为若干 chunk 后进行向量化检索。若切分边界恰好落在语义连续的句子或段落中间，检索到的 chunk 就缺乏足够上下文，导致生成答案时信息不完整、逻辑断裂，甚至引发幻觉。

---

## 主要解决方案

### 方案一：语义切分（Semantic Chunking）

**核心思路**
不再按固定字数/token 数机械切分，而是根据相邻句子的语义相似度来决定切分点。当两段相邻文本的嵌入距离超过阈值（即语义发生明显跳跃），才在此处切断，从而保证每个 chunk 内部语义完整。

**实现方法**
1. 将文档按句子分句；
2. 对每个句子（或固定大小的句子窗口）计算向量嵌入；
3. 计算相邻句子嵌入的余弦相似度；
4. 将相似度差异超过设定百分位数（默认 95th percentile）的位置作为切分点；
5. 按切分点合并句子形成 chunk。

**参考来源**
- [Weaviate: Chunking Strategies to Improve LLM RAG Pipeline Performance](https://weaviate.io/blog/chunking-strategies-for-rag)
- [LlamaIndex: Semantic Chunker 官方文档](https://developers.llamaindex.ai/python/examples/node_parsers/semantic_chunking/)
- [Springer Nature: Max–Min semantic chunking of documents for RAG application](https://link.springer.com/article/10.1007/s10791-025-09638-7)
- [Advanced RAG 05: Exploring Semantic Chunking](https://aiexpjourney.substack.com/p/advanced-rag-05-exploring-semantic-chunking-97c12af20a4d)

**推荐工具/框架**
- `LlamaIndex`: `SemanticSplitterNodeParser`
- `LangChain`: `SemanticChunker`（`langchain_experimental.text_splitter`）

---

### 方案二：带重叠的滑动窗口切分（Sliding Window with Overlap）

**核心思路**
在固定大小切分的基础上，让相邻 chunk 之间保留一定比例（通常 10%～20%）的重叠内容，使每个 chunk 的头尾都携带来自相邻 chunk 的上下文信息。

**实现方法**
1. 设置 `chunk_size`（如 512 tokens）和 `chunk_overlap`（如 64～128 tokens）；
2. 顺序滑动窗口生成 chunk，窗口每次前进 `chunk_size - chunk_overlap` 个 token；
3. 可结合句子边界检测（如 spaCy、NLTK）确保重叠区域不在句子中间断开。

**参考来源**
- [Unstructured: Chunking for RAG Best Practices](https://unstructured.io/blog/chunking-for-rag-best-practices)
- [NVIDIA Developer Blog: Finding the Best Chunking Strategy for Accurate AI Responses](https://developer.nvidia.com/blog/finding-the-best-chunking-strategy-for-accurate-ai-responses/)
- [Analytics Vidhya: 15 Chunking Techniques to Build Exceptional RAGs Systems](https://www.analyticsvidhya.com/blog/2024/10/chunking-techniques-to-build-exceptional-rag-systems/)

**推荐工具/框架**
- `LangChain`: `RecursiveCharacterTextSplitter(chunk_overlap=...)`
- `LlamaIndex`: `SentenceSplitter(chunk_overlap=...)`

### 方案三：父子层级检索（Parent Document Retriever / Small-to-Big）

**核心思路**
构建两层索引结构：用"小 chunk"（精细粒度）建立向量索引以提升检索精度，但实际返回给 LLM 的是对应"大 chunk"或父级文档（宽泛粒度）以保留完整上下文。从而同时兼顾检索精度和上下文完整性。

**实现方法**
1. 对同一文档同时创建"父 chunk"（大块，如整段或整节）和"子 chunk"（小块，如 2～3 句）；
2. 向量库仅存储子 chunk 的嵌入，但在元数据中记录父 chunk 的 ID；
3. 检索时使用子 chunk 做向量匹配，命中后回溯取出父 chunk 送入 LLM 作为上下文。

**参考来源**
- [LanceDB: Modified RAG: Parent Document & Bigger Chunk Retriever](https://lancedb.com/blog/modified-rag-parent-document-bigger-chunk-retriever-62b3d1e79bc6/)
- [GraphRAG: Parent-Child Retriever](https://graphrag.com/reference/graphrag/parent-child-retriever/)
- [AWS Bedrock Samples: Chunking Strategies for RAG Applications](https://aws-samples.github.io/amazon-bedrock-samples/rag/open-source/chunking/rag_chunking_strategies_langchain_bedrock/)

**推荐工具/框架**
- `LangChain`: `ParentDocumentRetriever`
- `LlamaIndex`: `SentenceWindowNodeParser` + `MetadataReplacementPostProcessor`

### 方案四：层级切分（Hierarchical Chunking）

**核心思路**
按文档原有的结构层级（章节 → 段落 → 句子）构建多层树状 chunk 索引。检索时可以从匹配的叶节点向上回溯扩展上下文，也可以从粗粒度节点向下定位精确内容，实现精度与召回的双重保障。

**实现方法**
1. 利用文档的结构标记（Markdown 标题、HTML 标签、PDF 大纲等）识别层级；
2. 按层级建立多级 chunk，每个 chunk 记录与父节点、兄弟节点的关系；
3. 检索命中叶节点后，系统自动拼接其所有祖先节点的摘要或全文，作为扩展上下文；
4. 可配合重排序模型（reranker）筛选最终送入 LLM 的内容。

**参考来源**
- [DataCamp: Chunking Strategies for AI and RAG Applications](https://www.datacamp.com/blog/chunking-strategies)
- [nb-data: 9 Chunking Strategies to Improve RAG Performance](https://www.nb-data.com/p/9-chunking-strategis-to-improve-rag)
- [Firecrawl: Best Chunking Strategies for RAG (and LLMs) in 2026](https://www.firecrawl.dev/blog/best-chunking-strategies-rag)

**推荐工具/框架**
- `LlamaIndex`: `HierarchicalNodeParser`
- 自定义实现：配合 `unstructured` 库提取文档结构

---

### 方案五：基于 LLM 的自适应切分（LLM-Based / Agentic Chunking）

**核心思路**
使用 LLM 本身来判断文档的切分点，而非依赖规则或相似度阈值。LLM 理解语义结构，能够识别"话题转换""论点完结"等语义边界，生成语义自洽、上下文完整的 chunk。

**实现方法**
1. 将文档分批送入 LLM，要求其输出合理的段落/切分建议（或直接输出 chunk 列表）；
2. 或采用"命题切分"（propositional chunking）：让 LLM 把每段提炼为独立的原子命题，以命题为单位建立索引；
3. 适合高价值文档，可离线预处理以节省成本。

**参考来源**
- [Databricks Technical Blog: The Ultimate Guide to Chunking Strategies for RAG Applications](https://community.databricks.com/t5/technical-blog/the-ultimate-guide-to-chunking-strategies-for-rag-applications/ba-p/113089)
- [LangCopilot: Document Chunking for RAG: 9 Strategies Tested (70% Accuracy Boost 2025)](https://langcopilot.com/posts/2025-10-11-document-chunking-for-rag-practical-guide)
- [Stack Overflow Blog: Breaking up is hard to do: Chunking in RAG applications](https://stackoverflow.blog/2024/12/27/breaking-up-is-hard-to-do-chunking-in-rag-applications/)

**推荐工具/框架**
- `LlamaIndex`: `AgenticChunker`（实验性）
- 自定义提示词 + GPT-4o / Claude 3.5 进行离线切分

## 推荐工具/框架汇总

| 工具/框架 | 支持的切分方式 | 适用场景 |
|---|---|---|
| LlamaIndex | SemanticSplitter、SentenceWindowNodeParser、HierarchicalNodeParser | 专注 RAG，功能最全 |
| LangChain | SemanticChunker、RecursiveCharacterTextSplitter、ParentDocumentRetriever | 通用 LLM 应用 |
| Unstructured | 结构化文档解析 + 层级切分 | PDF/HTML/Word 等文档 |

| Amazon Bedrock | 托管式 RAG，内置多种切分策略 | 云端一站式方案 |
