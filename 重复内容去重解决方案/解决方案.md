# 重复内容未去重——解决方案

## 问题描述（原文）

> 重复内容未去重，相同信息占据多个检索位

在 RAG 系统中，同一知识库往往存在大量重复或近似重复的内容（如同一文档的多个版本、爬取内容的重复段落、不同文档中相似的表述）。这些重复内容在索引时生成多个相近的向量，检索时被同时召回，占据有限的 Top-K 名额，导致：
- 多个检索位传递给 LLM 的实际信息量极低
- LLM 生成答案单调重复
- 真正多样的、有价值的上下文被挤出 Top-K
- 增加 Token 消耗，拖慢推理速度

---

## 主要解决方案

### 方案一：MinHash + LSH（局部敏感哈希）去重

**核心思路**

利用 MinHash 将文本文档压缩为固定长度的签名向量，再通过 LSH（Locality-Sensitive Hashing）将签名分 band 分桶，使相似文档落入同一桶中，从而以近线性复杂度检测海量文本中的近似重复对。

**实现方法**

1. 对文本做 n-gram（如 3-gram）切分，生成 shingle 集合
2. 用多个随机哈希函数对 shingle 集合做 MinHash，生成签名矩阵
3. 将签名按 band 分组，同 band 内相同的两个签名视为候选重复对
4. 对候选对计算精确 Jaccard 相似度，超过阈值（如 0.8）则确认为重复
5. 保留簇内代表文档，其余标记删除

**推荐库/工具**

- [`datasketch`](https://github.com/ekzhu/datasketch)：Python 实现，支持 MinHash LSH，易于集成
- [`text-dedup`](https://github.com/ChenghaoMou/text-dedup)：支持 MinHash、SimHash、Bloom Filter 多种去重方法
- [Milvus MinHash LSH](https://milvus.io/blog/minhash-lsh-in-milvus-the-secret-weapon-for-fighting-duplicates-in-llm-training-data.md)：向量数据库原生集成，适合大规模索引场景

### 方案二：SimHash 指纹去重

**核心思路**

SimHash 是一种快速的 LSH 算法，将文本映射为固定长度的二进制指纹，相似文本的 SimHash 汉明距离（Hamming Distance）很小。通过比较所有文档的 SimHash 指纹可快速识别近似重复。

**实现方法**

1. 对文本进行词频加权特征提取
2. 每个词的哈希向量按权重累加，正数取 1、负数取 0，生成 64-bit SimHash 指纹
3. 建立指纹索引，对新增文档检查汉明距离（阈值通常 ≤ 3）
4. 汉明距离在阈值内的文档判定为近重复，仅保留其一入库

**推荐库/工具**

- [`simhash`](https://github.com/leonsim/simhash)：轻量 Python 实现
- [`semhash`](https://minishlab.github.io/semhash-blogpost/)：针对 RAG 语义去重的工具，支持 embedding 级别相似度
- [GitHub: simhash-minhash-algorithm](https://github.com/RyanNSJ/simhash-minhash-algorithm)

**参考来源**

### 方案三：Maximal Marginal Relevance（MMR）检索时去重

**核心思路**

MMR 是一种检索后处理方法，在选取下一个文档时同时考虑：①与查询的相关性、②与已选文档集合的差异性，二者加权平衡，迭代贪婪地构建最终上下文集合，从源头上避免重复内容被一起传入 LLM。

**实现方法**

1. 检索 Top-K（较大值，如 40）候选文档
2. MMR 迭代：每次从候选集中选出使 `λ·sim(q,d) - (1-λ)·max_sim(d, selected)` 最大的文档加入已选集
3. 重复直到选出目标数量（如 6）的最终文档
4. λ 参数调节相关性与多样性权重（0.5 表示均衡，0.7 偏相关性，0.3 偏多样性）

**推荐库/工具**

- [LangChain MMR Retriever](https://python.langchain.com/docs/modules/data_connection/retrievers/mmr/)：原生支持 MMR
- [Azure AI Search MMR](https://farzzy.hashnode.dev/enhancing-rag-with-maximum-marginal-relevance-mmr-in-azure-ai-search)
- [Elasticsearch MMR](https://www.elastic.co/search-labs/blog/maximum-marginal-relevance-diversify-results)

### 方案四：多检索器融合后去重（Multi-Retriever Fan-Out Deduplication）

**核心思路**

在使用 Hybrid Retrieval（BM25 + 向量检索）或多检索器融合（RRF）时，同一内容段落可能来自多个检索器同时返回。在 RRF 合并、重排序（Cross-Encoder Rerank）之后，进行基于 ContentHash 或 embedding 余弦相似度的去重，既保留 RRF 排名信号，又消除冗余。

**实现方法**

1. 多路检索：BM25 + Dense Retrieval → 合并候选池
2. RRF 合并打分（保留排名信号）
3. Cross-Encoder 精排（提升精度）
4. 去重过滤：
   - 完全重复：ContentHash 相同 → 直接丢弃
   - 近似重复：embedding 余弦相似度 ≥ 0.90 → 保留排名更高者
5. 最终传入 LLM 的文档集合无重复

**推荐库/工具**

- [Kinde Retriever Fan-Out](https://www.kinde.com/learn/ai-for-software-engineering/using-ai-for-apis/retriever-fan-out-multi-retriever-rag-with-re-ranking-and-deduping/)：多检索器融合去重完整示例
- [Databricks RAG Data Pipeline](https://docs.databricks.com/aws/en/generative-ai/tutorials/ai-cookbook/quality-data-pipeline-rag)：企业级去重流水线
- Cohere Rerank、MS MARCO Cross-Encoder

### 方案五：索引阶段去重（Ingestion-time Deduplication Pipeline）

**核心思路**

在文档入库阶段建立去重管道，对每个新增 chunk 计算哈希/指纹，与现有索引对比，发现重复则跳过入库或合并元数据，从根本上防止重复内容进入向量库。

**实现方法**

1. 精确去重：对 chunk 文本计算 MD5/SHA-256，建立哈希集合，相同哈希直接跳过
2. 模糊去重：计算 MinHash/SimHash，与现有签名库对比，相似度超阈值则视为重复
3. 语义去重：对 embedding 向量进行 ANN（近似最近邻）查询，若最近邻距离 < 阈值，视为语义重复，不入库
4. 对重复文档记录来源信息，合并到已有 chunk 的元数据字段（来源多样性追踪）

**推荐库/工具**

- [semhash](https://minishlab.github.io/semhash-blogpost/)：专为 RAG 数据集去重设计
- [text-dedup](https://github.com/ChenghaoMou/text-dedup)：支持多种去重算法的一站式工具
