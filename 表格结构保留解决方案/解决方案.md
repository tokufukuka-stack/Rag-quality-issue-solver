# 表格数据被展平为纯文本后失去行列关系 — 解决方案

## 问题描述（原文）

> 表格数据被展平为纯文本后失去行列关系

在 RAG 系统中，PDF 或 HTML 文档中的表格往往在文档解析阶段被简单地转化为连续的纯文本字符串，导致原本清晰的行列语义结构（如"第3行第2列的值代表某产品在2023年的销量"）完全丢失。检索时，模型无法理解单元格之间的逻辑关系，严重影响问答质量。

---

## 解决方案

### 方案一：使用结构化格式（Markdown / HTML）保存表格

**核心思路：** 不将表格转为纯文本，而是将其整体序列化为 Markdown 或 HTML 格式，作为独立 chunk 存储。Markdown 的管道符（`|`）语法可以原生表达行列结构，LLM 能直接理解；HTML 则保留了更丰富的层级嵌套关系。

**实现方法：**
1. 在文档解析阶段，识别出表格区域，使用 `pdfplumber`、`Camelot`（PDF）或 `BeautifulSoup`（HTML）提取表格为 `pandas.DataFrame`。
2. 将 DataFrame 以 `df.to_markdown()` 或 `df.to_html()` 转换为 Markdown/HTML 字符串。
3. 将整张表格作为一个独立 chunk（而非按行切分）存入向量数据库，同时在 metadata 中记录表格标题、来源文档、页码等。
4. Embedding 时对表格 chunk 单独处理，可在内容前拼接表头描述以增强语义。

**参考来源：**
- [Beyond Plain Text: Egnyte's Journey to Structured Data Extraction in RAG Systems](https://www.egnyte.com/blog/post/beyond-plain-text-egnytes-journey-to-structured-data-extraction-in-rag-systems/)
- [How to handle tables during chunking - Rohan's Bytes](https://www.rohan-paul.com/p/how-to-handle-tables-during-chunking)
- [HTML to Markdown and Table Chunking Achieve 20% RAG Accuracy Gain - TATRAS DATA](https://tatrasdata.com/portfolio/html-to-markdown-and-table-chunking-achieve-20-rag-accuracy-gain/)

### 方案二：TableRAG — 专为表格理解设计的 RAG 框架

**核心思路：** TableRAG 将传统的全文检索替换为 **Schema 检索 + Cell 检索** 两级机制。Schema 检索帮助模型识别关键列名和数据类型，无需编码整张表；Cell 检索则定位包含关键值的具体单元格。两者结合后，大幅压缩 prompt 长度，同时减少信息丢失。

**实现方法：**
1. 对每张表格构建 Schema 索引（列名、类型、描述）和 Cell 索引（单元格内容 + 行列坐标）。
2. 查询时先通过 Schema 检索确定相关列，再通过 Cell 检索定位精确行。
3. 将检索到的局部表格片段（带行列标头）拼入 prompt，供 LLM 推理。
4. 支持对百万 token 级大表的高效问答，无需将完整表格塞入上下文。

**参考来源：**
- [TableRAG: Million-Token Table Understanding with Language Models (NeurIPS 2024, arXiv:2410.04739)](https://arxiv.org/abs/2410.04739)
- [TableRAG: A Retrieval Augmented Generation Framework for Heterogeneous Document Reasoning (EMNLP 2025)](https://aclanthology.org/2025.emnlp-main.710.pdf)
- [TableRAG 介绍 - MarkTechPost](https://www.marktechpost.com/2024/10/10/tablerag-a-retrieval-augmented-generation-rag-framework-specifically-designed-for-lm-based-table-understanding/)

### 方案三：表格自然语言摘要增强（Summary Augmentation）

**核心思路：** 对每张表格额外生成一段自然语言摘要（如"该表格列出了2020-2023年各地区销售额，最高值出现在华东地区2023年"），将摘要与结构化表格内容一并存入向量库。检索时摘要承担语义匹配，命中后返回原始结构化表格供 LLM 精确解读。

**实现方法：**
1. 解析阶段保留表格结构（Markdown/JSON 格式）。
2. 调用 LLM 对每张表格生成 1-3 句自然语言描述摘要。
3. 将"摘要文本"用于 embedding 并建立索引，将"原始结构化表格"存为关联内容（可存在 metadata 或独立字段）。
4. 检索时以摘要 embedding 匹配查询，返回完整结构化表格给生成模型。

**参考来源：**
- [Chunking Strategies for RAG: Best Practices and Key Methods - Unstructured](https://unstructured.io/blog/chunking-for-rag-best-practices)
- [Tabular Data, RAG, & LLMs: Improve Results Through Data Table Prompting (Intel Tech / Medium)](https://medium.com/intel-tech/tabular-data-rag-llms-improve-results-through-data-table-prompting-bcb42678914b)

### 方案四：深度学习表格检测与结构化提取（Table Transformer / Nougat）

**核心思路：** 对于扫描件、复杂 PDF 等规则解析效果差的场景，使用视觉模型（如 Microsoft Table Transformer、Meta Nougat）检测并识别表格区域，将其还原为结构化数据，再进行后续的格式化和存储。

**实现方法：**
1. 使用 Table Transformer（TATR）对文档图像进行表格检测和单元格识别。
2. 将识别结果转为 DataFrame，再序列化为 Markdown 或 JSON。
3. 对于学术文档，可直接使用 Nougat 模型将 PDF 整体转换为 LaTeX/Markdown，表格会被还原为 `\begin{tabular}` 或 Markdown 管道表格。
4. 输出结果可直接接入标准 RAG chunk 流程。

**参考来源：**
- [A Comparative Study of PDF Parsing Tools Across Diverse Document Categories (arXiv:2410.09871)](https://arxiv.org/html/2410.09871v1)
- [PdfTable: A Unified Toolkit for Deep Learning-Based Table Extraction (arXiv:2409.05125)](https://arxiv.org/html/2409.05125v1)
- [PDF parsing in RAG: How to parse PDF tables - Elasticsearch Labs](https://www.elastic.co/search-labs/blog/alternative-approach-for-parsing-pdfs-in-rag)

### 方案五：表格转知识图谱 / 关系三元组

**核心思路：** 将表格中的每个单元格关系抽象为结构化三元组（`<行头, 列头, 单元格值>`），存入知识图谱或结构化数据库（如 Neo4j、SQLite），使用自然语言查询时将其转为 SQL 或 SPARQL 查询，彻底绕过向量相似度检索的局限。

**实现方法：**
1. 解析表格后，将每行每列交叉关系表示为三元组，例如 `(产品A, 2023年销量, 1200万)`。
2. 存入图数据库（Neo4j）或关系型数据库（SQLite/PostgreSQL）。
3. 查询时使用 Text-to-SQL 或 Text-to-SPARQL 模型（如 TAPAS、DINSQL）将自然语言转为结构化查询。
4. 与向量 RAG 混合使用，文本问答走向量检索，表格问答走结构化查询。

**参考来源：**
- [RAG for Structured Data: Benefits, Challenges & Examples - AI21](https://www.ai21.com/knowledge/rag-for-structured-data/)
- [Beyond Extraction: Contextualising Tabular Data (arXiv:2401.02333)](https://arxiv.org/pdf/2401.02333)
