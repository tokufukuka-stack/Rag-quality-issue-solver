# 解决方案：向量相似度不等于语义相关度

## 问题描述（原文）

> 向量相似度不等于语义相关度，余弦相似高不代表真正有用

### 问题背景

在 RAG 系统中，文档检索通常依赖 embedding 模型将查询和文档编码为向量，再通过余弦相似度来排序候选结果。然而，这一方式存在根本性局限：

- **信息压缩损失**：文档被压缩为单一向量，大量细节信息丢失
- **分布偏差**：语义相近的词可能在向量空间中距离较远（如"贷款利率"与"借款成本"）
- **误配问题**：主题相近但对问题实际无用的文档可能获得高相似度分
- **否定/条件盲视**：无法区分"A 导致 B"与"A 不导致 B"的向量差异
- **粒度不对称**：查询通常很短，文档较长，压缩到同一维度后信息密度差异巨大

---

## 主要解决方案

### 方案一：两阶段检索 + 交叉编码器重排（Cross-Encoder Reranking）

**核心思路**

将检索分为两个阶段：第一阶段用 embedding 向量进行粗召回（速度快，保证 recall），第二阶段使用交叉编码器（cross-encoder）对查询与每个候选文档进行联合编码，输出精确的相关性分数，重新排序后取 Top-K。

**实现方法**

1. 向量检索召回 Top-50 ~ Top-100 候选文档
2. 将查询与每个候选文档拼接为 `[CLS] query [SEP] document [SEP]` 输入交叉编码器
3. 模型输出相关性分数（0~1 之间），按分数重新排列
4. 取最终 Top-K 送入 LLM 生成

**推荐工具/框架**

- **BGE Reranker（BAAI）**：`BAAI/bge-reranker-v2-m3`，支持多语言，性能优异
- **Cohere Rerank API**：托管服务，无需自部署
- **sentence-transformers**：本地部署，`cross-encoder/ms-marco-MiniLM-L-6-v2`
- **LlamaIndex / LangChain**：内置 reranker 节点支持

### 方案二：ColBERT 晚期交互模型（Late Interaction）

**核心思路**

ColBERT（Contextualized Late Interaction over BERT）改变了传统单向量压缩的方式：查询和文档分别保留 **token 级别的向量矩阵**，在检索时使用 MaxSim 算子对每个查询 token 找到文档中最相似的 token，再求和得到最终相关性分数。这种方式在速度与精度之间取得了更好的平衡，且对语义细节捕捉能力远强于单向量。

**实现方法**

1. 离线对文档做 token 级别编码，存入 ColBERT 索引
2. 查询时编码查询的每个 token
3. 用 MaxSim 算子计算查询向量矩阵与文档向量矩阵的相似度
4. 对候选文档打分并重排

**推荐工具/框架**

- **RAGatouille**：最简单的 ColBERT Python 集成库（`pip install ragatouille`）
- **Stanford ColBERT**：原始实现（[github.com/stanford-futuredata/ColBERT](https://github.com/stanford-futuredata/ColBERT)）
- **Jina ColBERT v2**：多语言版本，支持 89 种语言，含中文
- **IBM Developer ColBERT 教程**：[How ColBERT works in RAG | IBM](https://developer.ibm.com/articles/how-colbert-works/)

### 方案三：混合检索（Hybrid Search：Dense + Sparse）

**核心思路**

单一的向量检索容易遗漏关键词精确匹配的文档，而 BM25 等稀疏检索擅长精确词汇匹配但无法处理语义变体。**混合检索**结合两种方式，通过 RRF（Reciprocal Rank Fusion）或加权融合将两个排序列表合并，显著提升整体检索质量。

在混合检索中进一步引入 **SPLADE**（Sparse Lexical and Expansion Model）可以同时实现词汇精确性和查询扩展能力，弥补纯余弦相似度的不足。

**实现方法**

1. 分别用 BM25/SPLADE（稀疏）和 embedding 模型（密集）检索各取 Top-K
2. 使用 RRF 公式合并排名：`score(d) = Σ 1/(k + rank_i(d))`
3. 可选：在融合结果基础上再用 cross-encoder 精排

**推荐工具/框架**

- **Elasticsearch / OpenSearch**：内置 BM25 + kNN 混合检索
- **Weaviate / Qdrant**：支持 hybrid search 原生 API
- **SPLADE**：[Pinecone SPLADE 教程](https://www.pinecone.io/learn/splade/)
- **LangChain EnsembleRetriever**：最简单的 Python 混合检索实现

**参考来源**

### 方案四：LLM 辅助相关性评分（LLM-as-Reranker）

**核心思路**

直接调用 LLM（如 GPT-4、Claude）对每个候选文档进行相关性判断，借助大模型的深层语义理解能力超越向量相似度的限制。典型实现包括：
- **RankGPT**：让 LLM 生成候选文档的排名列表
- **PointWise scoring**：对每对（查询, 文档）打分（0~10）
- **ListWise**：给 LLM 提供候选列表，要求输出重排后的顺序

**实现方法**

```python
# 示例：PointWise LLM Reranker
prompt = f"""Rate the relevance of the following document to the query on a scale of 0-10.
Query: {query}
Document: {doc}
Score (0-10):"""
score = llm.generate(prompt)
```

**推荐工具/框架**

- **LlamaIndex LLMRerank**：内置 LLM 重排节点
- **RankGPT**：[GitHub: sunnweiwei/RankGPT](https://github.com/sunnweiwei/RankGPT)
- **FlashRank**：轻量快速重排库，可选 LLM 后端

### 方案五：查询感知文档摘要 + 重嵌入（Query-aware Summarization / HyDE）

**核心思路**

在检索前对查询进行增强，使其向量更接近真正相关文档的向量分布：
- **HyDE（Hypothetical Document Embeddings）**：让 LLM 根据查询先生成一个"假设性答案文档"，再用该文档的 embedding 进行检索，显著缩小查询向量与目标文档向量之间的语义鸿沟
- **Query Expansion**：通过 LLM 扩展查询词，生成多个变体，合并检索结果

**实现方法**

```python
# HyDE 实现示意
hypothetical_doc = llm.generate(f"Write a document that answers: {query}")
embedding = embed_model.encode(hypothetical_doc)
results = vector_db.search(embedding, top_k=10)
```

**推荐工具/框架**

- **LlamaIndex HyDEQueryTransform**：内置 HyDE 支持
- **LangChain HypotheticalDocumentEmbedder**：封装好的 HyDE 实现
- **MultiQueryRetriever**：生成多个查询变体并合并结果
