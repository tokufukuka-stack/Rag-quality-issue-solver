# RAG 故障排查定位：检索端 vs 生成端问题归因

## 问题描述（原文）

> 故障排查困难——问题出在检索端还是生成端难以定位

在 RAG 系统中，当最终输出出现问题（如答案错误、幻觉、不相关等），工程师很难判断故障根源是：
- **检索端**：召回了错误/不相关的文档，或漏召回了关键文档
- **生成端**：LLM 没有正确利用已检索到的正确文档，凭训练知识编造了内容

这两类问题的修复策略完全不同，混淆诊断会导致资源浪费与效果退化。

---

## 解决方案

### 方案一：RAG 三元组评估框架（RAG Triad / RAGAS 指标体系）

**核心思路**

将 RAG 管道的评估分解为三个独立维度，通过量化每个维度的得分来实现故障定位：
1. **Context Relevance（上下文相关性）**：检索到的文档与用户问题是否相关 → 检索端质量
2. **Groundedness / Faithfulness（忠实度）**：生成的答案是否基于检索内容 → 生成端质量
3. **Answer Relevance（答案相关性）**：最终答案是否回答了用户的问题 → 端到端质量

**诊断逻辑**

| 场景 | Context Relevance | Faithfulness | 故障定位 |
|------|------------------|--------------|---------|
| 检索失败 | 低 | 高/低 | 检索端问题 |
| 生成幻觉 | 高 | 低 | 生成端问题 |
| 两端均正常 | 高 | 高 | 其他原因（prompt 设计等）|
| 两端均失败 | 低 | 低 | 需分别修复 |

**实现方法**

使用 [RAGAS](https://github.com/explodinggradients/ragas) 框架：
```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

result = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)
```

### 方案二：组件级隔离诊断框架（RAG-X / RAGVue）

**核心思路**

通过设计受控实验，将检索器和生成器解耦独立评估：
- **固定完美上下文，测试生成器**：手动提供已知正确的文档片段，测试 LLM 是否能生成正确答案。若失败 → 生成端问题
- **固定 LLM，测试检索器**：使用 oracle 检索（直接给 LLM 标准答案文档），若 LLM 能正确回答 → 检索端问题

**实现方法**

参考 RAG-X 多维度评估框架，对以下维度独立打分：
- **Coverage（覆盖度）**：检索结果是否包含答案所需信息
- **Redundancy（冗余度）**：检索结果中无关信息的比例
- **Faithfulness（忠实度）**：生成内容与检索内容的一致程度
- **Reasoning Adequacy（推理充分度）**：LLM 是否正确推理利用了上下文

```python
# 伪代码示例：组件隔离测试
def test_retriever_only(query, ground_truth_docs):
    """固定理想文档，测试生成器"""
    answer = llm.generate(query, context=ground_truth_docs)
    return evaluate_answer(answer, expected)

def test_generator_only(query, retrieved_docs):
    """使用检索结果，测试检索器质量"""
    relevance_score = score_retrieval(retrieved_docs, query)
    return relevance_score
```

### 方案三：全链路可观测性追踪（LangSmith / Arize Phoenix）

**核心思路**

在 RAG 管道中集成 OpenTelemetry 标准的追踪系统，记录每个组件的输入/输出、延迟和中间状态，通过可视化界面快速定位故障节点。

**实现方法**

以 LangSmith 为例：
```python
from langsmith import traceable

@traceable(name="retrieval_step")
def retrieve(query: str) -> list:
    # 检索逻辑
    return vectorstore.similarity_search(query)

@traceable(name="generation_step")
def generate(query: str, context: list) -> str:
    # 生成逻辑
    return llm.invoke(f"Context: {context}\nQuery: {query}")
```

集成 Arize Phoenix（开源）：
```python
import phoenix as px
from openinference.instrumentation.langchain import LangChainInstrumentor

px.launch_app()
LangChainInstrumentor().instrument()
```

通过 UI 界面可以看到：
- 每次查询的检索文档及其相关性分数
- LLM 的 prompt 内容和生成内容
- 各步骤的执行时间和 token 消耗

**参考来源**

### 方案四：交互式实时调试工具（RAGGY / RAG Without the Lag）

**核心思路**

采用交互式调试范式，允许工程师在调试过程中实时干预和修改检索结果或 prompt，快速验证假设，缩短 Debug 循环。RAGGY 系统提供可组合的 RAG 原语库与交互式界面。

**实现方法**

核心调试步骤：
1. **可视化中间结果**：展示检索到的每个文档片段及其相似度分数
2. **手动注入/替换**：允许工程师手动替换检索结果，重新运行生成步骤
3. **对比实验**：同一问题，分别用"理想检索结果"和"实际检索结果"对比生成质量
4. **归因追溯**：对生成输出中的每句话，追溯其是否有检索文档作为依据

**参考来源**
- RAG Without the Lag 论文（Arxiv 2025）：https://arxiv.org/abs/2504.13587
- Maxim RAG 调试指南：https://www.getmaxim.ai/articles/rag-debugging-identifying-issues-in-retrieval-augmented-generation/
- DEV Community RAG Debugging Playbook：https://dev.to/kuldeep_paul/the-rag-debugging-playbook-a-step-by-step-guide-to-trace-level-failures-and-fixes-56pa

### 方案五：分层幻觉检测与归因（RT4CHART / Datadog LLM Observability）

**核心思路**

通过将生成答案拆解为原子级别的声明（claims），对每个声明进行独立验证，精确识别哪些声明有检索支撑、哪些是幻觉，从而将故障精确归因到具体文档或生成行为。

**实现方法**

层次化验证流程：
1. **Answer Decomposition（答案分解）**：将生成答案拆分为独立的事实声明
2. **Local Verification（局部验证）**：每条声明与对应检索块进行一致性检验
3. **Global Verification（全局验证）**：在全部检索内容范围内重新验证
4. **Attribution Score（归因分数）**：计算每条声明的来源可信度

使用 Datadog LLM Observability：
```python
from ddtrace.llmobs import LLMObs
LLMObs.enable(ml_app="rag_app")

# 自动追踪幻觉并归因到检索或生成组件
```
