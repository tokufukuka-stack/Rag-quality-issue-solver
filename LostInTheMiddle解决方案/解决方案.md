# "Lost in the Middle" 效应解决方案

## 问题描述（原文）

> "lost in the middle"效应——模型倾向关注上下文头尾，忽略中间内容

### 问题背景

2023 年，Stanford/Salesforce 等机构的研究者 Nelson F. Liu 等人在论文 *Lost in the Middle: How Language Models Use Long Contexts*（ACL 2024 TACL）中首次系统性地揭示了这一现象：当向 LLM 输入包含多段文档的长上下文时，模型在回答问题时对位于上下文**开头**和**末尾**的信息利用率显著高于**中间**位置的信息，性能呈 U 形曲线分布。在多文档问答任务中，当相关信息从首尾移到中间位置时，模型准确率可下降超过 30%。

**根本原因**：主流 LLM 使用的旋转位置编码（RoPE）天然存在"长距离衰减"效应——随着位置距离增大，注意力权重递减，导致模型对中间 token 的关注度系统性偏低；同时，自注意力机制中的首因效应（primacy）和近因效应（recency）也强化了这一偏向。

---

## 主要解决方案

### 方案一：策略性文档重排（Position-Aware Reranking）

**方案名称**：LostInTheMiddleReranker / 首尾优先排列

**核心思路**：将检索到的文档按相关性打分后，把得分最高的文档放置在上下文的**开头和末尾**，得分较低的文档放置在中间，充分利用模型的首因与近因注意力偏好。

**实现方法**：
1. 使用 Cross-Encoder 重排模型（如 `cross-encoder/ms-marco-MiniLM-L-6-v2`、`BGE-Reranker`）对所有候选文档打分
2. 按相关性排序后，采用"交错放置"策略：将 rank-1 放首位，rank-2 放末位，rank-3 放次首位，以此类推
3. LlamaIndex 内置了 `LostInTheMiddleReranker` 类，可直接调用

**参考来源**：

### 方案二：提示词压缩（Prompt Compression via LongLLMLingua）

**方案名称**：LongLLMLingua 提示压缩

**核心思路**：在检索后生成前，对检索到的文档片段进行信息密度压缩，删除冗余内容，保留关键信息，并将最核心的内容重新调度到上下文的前部，从而缩短整体上下文长度并提升关键信息的可见性。

**实现方法**：
1. 检索阶段正常执行，获取 Top-K 文档
2. 调用 LongLLMLingua（Microsoft 开源）对检索结果进行细粒度压缩
3. 流程：重排（Reranking）→ 细粒度提示压缩（Fine-Grained Prompt Compression）→ 子序列恢复（Subsequence Recovery）
4. 压缩率可达 4:1（使用原上下文 25% 的 token），同时准确率提升最高可达 21.4%

**参考来源**：
- [LongLLMLingua LlamaIndex 官方博客](https://www.llamaindex.ai/blog/longllmlingua-bye-bye-to-middle-loss-and-save-on-your-rag-costs-via-prompt-compression-54b559b9ddf7)
- [LongLLMLingua Medium 详解](https://medium.com/llamaindex-blog/longllmlingua-bye-bye-to-middle-loss-and-save-on-your-rag-costs-via-prompt-compression-54b559b9ddf7)

### 方案三：多尺度位置编码（Ms-PoE）

**方案名称**：Multi-scale Positional Encoding (Ms-PoE) — "Found in the Middle"

**核心思路**：从模型架构层面解决问题，无需微调。通过对不同注意力头（attention heads）施加**不同的位置索引缩放比例**，使得各注意力头同时覆盖短程和长程依赖，从而"解除"RoPE 长距离衰减对中间位置的抑制。

**实现方法**：
1. 引入论文 *Found in the Middle: How Language Models Use Long Contexts Better via Plug-and-Play Positional Encoding*（Arxiv 2403.04797）提出的 Ms-PoE 方法
2. 将模型的注意力头分组，对不同组施加不同的位置缩放系数（scaling ratio）
3. 该方法为 plug-and-play，无需额外训练，仅需在推理时修改位置嵌入计算逻辑
4. 在多文档 QA 和 KV 检索任务上，中间位置信息提取准确率可提升 20–40%

**参考来源**：
- [Found in the Middle 论文 (Arxiv 2403.04797)](https://arxiv.org/abs/2403.04797)
- [Found in the Middle 论文全文](https://arxiv.org/html/2403.04797v1)
- [Lost in the Middle 原始论文 (Arxiv 2307.03172)](https://arxiv.org/abs/2307.03172)

### 方案四：上下文感知检索 + 文档压缩（Contextual Retrieval + Compression）

**方案名称**：Anthropic Contextual Retrieval + LangChain ContextualCompressionRetriever

**核心思路**：通过两项手段共同降低需要送入 LLM 的上下文总量：(1) 在切片时为每个 chunk 自动添加文档级上下文（减少语境丢失）；(2) 检索后使用文档压缩器，仅保留每个 chunk 中真正与 query 相关的片段，缩减总 token 数，使关键信息集中在上下文前端。

**实现方法**：
1. **Contextual Retrieval**（Anthropic）：切片时使用 LLM 为每个 chunk 生成一段"上下文摘要前缀"并拼接，保留文档级信息
2. **ContextualCompressionRetriever**（LangChain）：将 `BaseRetriever` 与 `DocumentCompressor` 组合，检索后压缩每个文档至与查询最相关的句子或段落
3. 两者结合使用，可在缩短上下文的同时保留语义完整性
4. 进一步可结合 GraphRAG 的 query-focused summarization

**参考来源**：
- [Anthropic Contextual Retrieval 博客](https://www.anthropic.com/news/contextual-retrieval)
- [Neo4j Advanced RAG Techniques](https://neo4j.com/blog/genai/advanced-rag-techniques/)
- [Stack Overflow RAG Practical Tips](https://stackoverflow.blog/2024/08/15/practical-tips-for-retrieval-augmented-generation-rag/)

### 方案五：两阶段检索 + 精简 Top-K（Two-Stage Retrieval with Strict Top-K）

**方案名称**：Broad Recall + Precision Rerank + 严格限制 Top-K

**核心思路**：通过控制最终送入 LLM 的文档数量（保持 3–5 个），从根本上缩短上下文长度，使所有相关信息都能落在模型注意力较强的区域（首尾），彻底规避中间盲区。

**实现方法**：
1. 第一阶段：使用混合检索（Dense + BM25 Hybrid Search）进行宽松召回，获取 20–100 个候选文档
2. 第二阶段：使用 Cross-Encoder 精排（如 Cohere Rerank、BGE-Reranker-v2、Voyage AI Reranker）将候选缩减至精准 Top-3~5
3. 仅将 Top-3~5 文档送入 LLM，避免长上下文问题
4. 搭配 MapReduce 或 Refine 模式处理确实需要大量文档的场景
