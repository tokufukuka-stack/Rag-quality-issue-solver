# 可行性分析：层级结构丢失解决方案

> 本文档对五种解决方案进行独立可行性评估，分析维度包括技术可行性、资源需求、预期效果、潜在风险，并给出综合推荐。

---

## 方案一：分层父子 Chunk 策略（Hierarchical Parent-Child Chunking）

### 技术可行性
**实现难度：低**

LangChain 和 LlamaIndex 均提供开箱即用的组件（`ParentDocumentRetriever`、`HierarchicalNodeParser` + `AutoMergingRetriever`），无需从头实现。只需配置父/子 chunk 大小参数即可接入现有 RAG 流程。

### 资源需求
- **算力**：仅需在存储层维护双重索引（父文档 docstore + 子文档 vectorstore），增量内存开销约为原始向量索引的 2-3 倍。
- **数据**：无需额外标注数据，直接基于原始文档构建。
- **人力**：低，工程师配置时间约 1-2 天即可集成到现有系统。

### 预期效果评估
- 对有清晰章节结构的文档（技术文档、学术论文、法规文本）效果显著，召回质量可提升 10-25%。
- 对无结构或结构混乱的文档（扫描件、聊天记录）效果有限，层级关系本身不存在时父子 chunk 也无从构建。
- 在 AutoMerging 场景下，可减少碎片化 chunk 导致的上下文缺失问题。

### 潜在风险与局限
- 父 chunk 过大可能引入噪声，导致 LLM 注意力分散（与 "lost in the middle" 问题交叉）。
- 需要合理设定父子比例（通常父:子 = 4:1），参数敏感，不同文档类型最优值不同。
- docstore 与 vectorstore 双重存储增加了系统复杂度和维护成本。

### 综合推荐指数
⭐⭐⭐⭐½（4.5/5）

### 小结
工程落地最简单、收益最直接，是解决层级结构丢失问题的**首选方案**，适合大多数有结构的文档场景。

---

## 方案二：结构感知文档解析 + 层级元数据注入（Metadata Breadcrumbs）

### 技术可行性
**实现难度：中**

Markdown 格式文档可直接使用 `MarkdownHeaderTextSplitter`，较容易实现。但对于 PDF、Word 等原始格式文档，需要先进行格式转换（PDF→Markdown），引入额外的转换步骤和转换质量风险。

### 资源需求
- **算力**：元数据存储和过滤基本零额外算力开销；格式转换（如使用 Docling）需要 CPU/GPU 时间，与文档规模成正比。
- **数据**：无需标注数据，但需要建立文档标题层级的解析规则（可能需要针对特定文档格式手动配置）。
- **人力**：中等，约需 3-5 天完成格式转换流程接入与元数据 schema 设计。

### 预期效果评估
- 对格式规范的 Markdown/HTML 文档效果极佳，层级路径信息可支持精确的结构化过滤。
- 面包屑元数据可作为 Hybrid Search 的过滤维度，显著提升精确检索场景（如"找第三章第二节的内容"）的准确率。
- 对格式不规范、标题级别混乱的文档（如手动排版的 PDF），解析层级准确率可能不稳定。

### 潜在风险与局限
- PDF 到 Markdown 的转换本身可能出错，标题识别失败导致层级信息不准确。
- 元数据字段的 schema 设计需要一定经验，否则过滤逻辑难以通用化。
- 对查询方式有要求：用户查询需要明确涉及结构（如"第二章"）才能充分利用元数据过滤的优势。

### 综合推荐指数
⭐⭐⭐⭐（4/5）

### 小结
是**最具可扩展性**的方案，适合将结构化检索能力系统化集成，建议与方案一联合使用（父子 chunk + 元数据过滤）。

## 方案三：HiQA 层级上下文增强（Hierarchical Contextual Augmentation）

### 技术可行性
**实现难度：中**

HiQA 的核心思路（级联标题元数据 + Markdown 格式化）可以自行工程复现，不依赖专有框架。但其 Multi-Route Retriever 部分需要自定义开发，且论文中的完整实现未开源，需要根据论文自行复现。

### 资源需求
- **算力**：为每个 chunk 生成 augmented text（将层级路径前缀拼接到 chunk 文本）并重新 embedding，embedding 计算量约增加 10-30%（因 token 长度增加）。
- **数据**：Markdown 转换需要处理全量文档库，初期有一定的转换成本。
- **人力**：中-高，复现 HiQA 完整流程约需 1-2 周工程投入。

### 预期效果评估
- 该方法在 embedding 层面直接将层级信息融入向量表示，理论上能提升语义检索时对结构的感知能力。
- 实验表明，在多文档 QA 场景下，HiQA 相比基线 RAG 在准确率上有显著提升（论文报告显示在某些基准上提升 5-15%）。
- 适合需要跨文档、跨章节综合推理的复杂问答场景。

### 潜在风险与局限
- Augmented embedding 文本的标题路径过长时，可能超出 embedding 模型的 token 限制，需要截断处理。
- 将结构信息融入 embedding 会改变向量空间的分布，可能对某些原本良好的检索场景产生负向影响（需要实验验证）。
- 论文未完整开源代码，工程复现存在不确定性。

### 综合推荐指数
⭐⭐⭐½（3.5/5）

### 小结
理论上效果最好，但工程复现成本较高。建议作为**进阶优化方案**，在基础方案（方案一+二）稳定后再考虑实施。

## 方案四：递归文档结构解析（Recursive Document Parsing）

### 技术可行性
**实现难度：低**

`RecursiveCharacterTextSplitter` 是 LangChain 的标准组件，配置分隔符列表即可按标题层级递归切分。对于 Markdown 文档几乎零额外开发。

### 资源需求
- **算力**：极低，纯文本处理，无需 GPU，切分速度极快。
- **数据**：无需额外数据，但文档需要有规范的 Markdown 格式。
- **人力**：极低，约半天即可完成配置和测试。

### 预期效果评估
- 对于已经是 Markdown 格式的文档（如技术文档、README 文件、Wiki 页面），效果良好，能按自然章节边界切分，避免句中截断。
- 相比纯固定大小切分，结构感知切分可使每个 chunk 语义更完整，向量表示质量更高。
- 效果弱于方案一（因为没有父子层级的回溯机制），适合作为基础方案或与方案一结合。

### 潜在风险与局限
- 仍然只是改善了切分位置，没有在 chunk 元数据中显式记录层级信息，无法支持基于结构的过滤检索。
- 对于无规范结构的文档（纯文本、扫描件转文字），效果与普通递归切分无异。
- 某些极长的章节（如单一章节数万字）仍需在章节内再次切分，层级信息在二次切分时丢失。

### 综合推荐指数
⭐⭐⭐（3/5）

### 小结
实现成本最低，是**快速起步的入门方案**，但层级信息保留不彻底，适合资源有限或文档格式规范的场景，建议后续升级到方案一或二。

## 方案五：视觉文档理解（Vision-Guided Document Parsing）

### 技术可行性
**实现难度：高**

需要集成专用的文档理解模型（VLM 或 Document AI），涉及 GPU 推理服务部署或云 API 调用，技术栈复杂度显著提升。Azure Document Intelligence 提供托管 API 可降低部署难度，但引入外部服务依赖。

### 资源需求
- **算力**：高。本地部署 VLM（如 LayoutLM、Nougat）需要 GPU 资源；使用 Azure API 则按页计费（约 $0.001-0.01/页），大规模文档库成本可观。
- **数据**：本地微调需要带结构标注的文档数据集；API 方案无需数据。
- **人力**：高，需要熟悉 Document AI 的工程师，接入和调试成本约 1-3 周。

### 预期效果评估
- 对于 PDF、扫描件等非文本格式文档，是效果最好的方案，能够识别视觉上的标题（大字体、加粗）即使文档本身没有语义标记。
- 对于格式复杂的报告（含图表、多栏布局、页眉页脚），可以更准确地提取内容结构。
- 理论上可以同时解决层级结构丢失 + 多模态内容无法索引两个问题，一举两得。

### 潜在风险与局限
- API 成本高，大批量文档处理成本不可忽视（百万页文档可能花费数千美元）。
- 本地部署 VLM 资源需求大，且模型推理速度慢（相比纯文本解析慢 10-100 倍）。
- 对于已有完整 Markdown/HTML 格式的文档，此方案属于"大炮打蚊子"，性价比低。
- 结构识别仍有错误率，尤其面对设计风格特殊的文档。

### 综合推荐指数
⭐⭐⭐（3/5）

### 小结
适合**特定高价值场景**（如大量 PDF 报告、扫描档案），但对于大多数 RAG 系统而言成本过高。建议仅在文档格式复杂且无结构化文本版本时采用，或仅对关键文档批次使用。

## 最终推荐方案

### 推荐组合：方案一 + 方案二 联合使用

**理由：**

1. **方案一（父子 Chunk 分层索引）** 解决了"检索到的 chunk 缺乏上下文"的核心问题，通过在返回时自动扩展父 chunk 来补充语境，实现成本低，收益明显。

2. **方案二（结构元数据面包屑注入）** 在方案一的基础上进一步引入可过滤的结构元数据，支持"精确定位到章节"的结构化检索需求，将层级信息从隐式（通过父 chunk 传递）变为显式（通过元数据查询）。

3. 两者结合后，既能在语义检索层面感知层级（父子 chunk），又能在结构化过滤层面利用层级（元数据 breadcrumb），形成互补。

**实施路径建议：**

1. **第一步（1-2天）**：将文档转换为 Markdown 格式（使用 Docling 或 Marker），确保标题层级保留。
2. **第二步（1-2天）**：使用 `MarkdownHeaderTextSplitter` 切分并注入层级元数据，建立新的向量索引。
3. **第三步（1-2天）**：引入 `ParentDocumentRetriever`（LangChain）或 `HierarchicalNodeParser + AutoMergingRetriever`（LlamaIndex），构建父子双层索引。
4. **第四步（持续）**：针对具有复杂 PDF 格式的核心文档，考虑引入方案五（Azure Document Intelligence）进行更精确的结构提取。

**预期总体改善效果：** 结构化文档的检索准确率提升 15-30%，用户对"找不到相关章节内容"问题的反馈减少。
