# RAG 数据泄露防护解决方案

## 问题描述（原文）

> 数据泄露——检索过程可能将敏感文档暴露给无权限用户

在 RAG 系统中，检索阶段会从向量数据库中召回与用户查询最相关的文档片段。若缺乏有效的访问控制机制，恶意用户可通过精心设计的提示词提取出本无权访问的敏感内容，甚至可以通过向量逆向工程（embedding inversion attack）重建原始私密文档。这一问题在企业多租户场景、医疗、法律、金融等高度敏感领域中尤为突出。

---

## 主要解决方案

### 方案一：检索层访问控制（Retrieval-Time Access Control）

**核心思路**

在向量检索阶段，将用户身份/角色信息作为元数据过滤条件注入查询中，仅召回当前用户有权访问的文档块（chunk）。

**实现方法**

1. **文档索引时**：为每个 chunk 附加访问控制元数据（如 `allowed_roles: ["admin", "finance"]` 或 `allowed_users: ["user_123"]`）。
2. **检索时**：在向量相似度查询中叠加元数据过滤条件，确保只有符合权限的 chunk 进入候选集。
3. **数据库支持**：Pinecone 的 metadata filter、Milvus 的 Row-Level RBAC、Supabase pgvector 的 Row Level Security（RLS）、Weaviate 的 `where` filter 均原生支持此模式。

**参考来源**

- [RAG with Access Control | Pinecone](https://www.pinecone.io/learn/rag-access-control/)
- [RAG with Permissions | Supabase Docs](https://supabase.com/docs/guides/ai/rag-with-permissions)
- [Enabling Fine-Grained Access Control with Milvus Row-Level RBAC | Zilliz](https://zilliz.com/blog/enabling-fine-grained-access-control-with-milvus-row-level-rbac)
- [The Right Approach to Authorization in RAG | OSO](https://www.osohq.com/post/right-approach-to-authorization-in-rag)

### 方案二：细粒度授权策略引擎（Fine-Grained Authorization Engine）

**核心思路**

引入独立的授权策略引擎（如 Cerbos、OPA、Auth0 FGA），将访问控制逻辑从业务代码中解耦，在向量库查询之前/期间动态生成过滤条件（Query Plan），实现基于策略的文档级访问控制。

**实现方法**

1. 使用 Cerbos 或 Auth0 FGA 定义资源访问策略（哪些角色可以访问哪些文档分类）。
2. 在 RAG 管道的检索步骤前，调用授权引擎生成查询过滤计划（authorization query plan）。
3. 将该过滤计划转化为向量数据库的元数据过滤参数，传入检索请求。
4. 检索结果在进入 LLM 上下文前再做一次后置校验。

**参考来源**

- [Secure RAG: Implement LLM Access Control With Cerbos](https://www.cerbos.dev/blog/access-control-for-rag-llms)
- [Build Trustworthy AI: Implementing Access Control for RAG Using FGA | Auth0](https://auth0.com/blog/rag-and-access-control-where-do-you-start/)
- [Securing Agentic/RAG Pipelines with Fine-Grained Authorization | Couchbase](https://www.couchbase.com/blog/securing-agentic-rag-pipelines/)

### 方案三：PII 检测与脱敏（PII Detection & Redaction）

**核心思路**

在文档入库（索引阶段）和/或检索结果返回给 LLM 之前，使用 NLP 工具自动识别并脱敏个人身份信息（PII），防止敏感实体（姓名、身份证、电话、邮件等）直接暴露在生成上下文中。

**实现方法**

1. **入库时脱敏**：集成 Microsoft Presidio 或 Amazon Comprehend，对文档进行 NER + 正则混合扫描，用占位符替换 PII（如 `<PERSON>`, `<EMAIL>`）后再入库。
2. **检索后脱敏**：对召回的 chunk 做实时 PII 过滤，再送入 LLM。
3. **Amazon Bedrock** 提供托管式 RAG 敏感数据保护：自动触发 Comprehend PII 检测任务，支持 Lambda 函数驱动的流水线自动化。
4. 使用合成数据替换真实 PII，保留上下文语义同时避免泄露。

**参考来源**

- [PII Detector for RAG: Presidio Masking Guide | LlamaIndex](https://www.llamaindex.ai/blog/pii-detector-hacking-privacy-in-rag)
- [Protect sensitive data in RAG applications with Amazon Bedrock | AWS](https://aws.amazon.com/blogs/machine-learning/protect-sensitive-data-in-rag-applications-with-amazon-bedrock/)
- [Guidance for Securing Sensitive Data in RAG Applications using Amazon Bedrock | AWS Solutions](https://aws-solutions-library-samples.github.io/ai-ml/securing-sensitive-data-in-rag-applications-using-amazon-bedrock.html)

### 方案四：差分隐私（Differential Privacy for RAG）

**核心思路**

在检索或聚合阶段引入差分隐私（DP）噪声机制，使攻击者无法通过观察输出推断出某单个私密文档是否参与了检索，从而在数学上保证隐私性。

**实现方法**

1. **DP-RAG 框架**：基于 token-by-token 聚合策略，对多个文档的检索结果分别生成 token 分布，加入高斯/Laplace 噪声后再聚合输出，避免直接暴露单一文档内容。
2. **局部差分隐私（LDP）**：在实体级别对敏感命名实体进行扰动，再送入 LLM，兼顾可用性与隐私保护。
3. **开源实现**：`sarus-tech/dp-rag`（GitHub）提供了 DP-RAG 的简单参考实现。

**参考来源**

- [RAG with Differential Privacy | Arxiv 2412.19291](https://arxiv.org/html/2412.19291v1)
- [Mitigating privacy risks in RAG via locally private entity perturbation | ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/S0306457325000913)
- [GitHub - sarus-tech/dp-rag](https://github.com/sarus-tech/dp-rag)

### 方案五：加密检索与可证明安全 RAG（Provably Secure RAG）

**核心思路**

在文档入库前对文本内容及其对应 embedding 向量同时加密存储；检索时进行验证与解密，确保即使向量数据库本身遭到攻击，攻击者也无法还原原始文档内容。

**实现方法**

1. **SAG 框架（Secure Augmented Generation）**：入库时加密文本+embedding，检索时解密验证，兼容现有 RAG 管道，提供机密性与完整性双重保障。
2. **Privacy-Aware RAG**：知识图谱级别的隐私保护架构，在实体/关系层面引入隐私感知处理器，支持元素级细粒度隐私控制。
3. 利用同态加密或安全多方计算（MPC）对向量检索过程进行加密处理（学术探索阶段，工程成本高）。

**参考来源**

- [Provably Secure Retrieval-Augmented Generation | Arxiv 2508.01084](https://arxiv.org/html/2508.01084v1)
- [Privacy-Aware RAG: Secure and Isolated Knowledge Retrieval | Arxiv 2503.15548](https://arxiv.org/html/2503.15548v1)
- [The Good and The Bad: Exploring Privacy Issues in RAG | ACL 2024](https://aclanthology.org/2024.findings-acl.267/)
