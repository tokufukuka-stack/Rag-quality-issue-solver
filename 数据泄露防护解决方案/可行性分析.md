# 可行性分析：RAG 数据泄露防护方案

> 本文档对「数据泄露——检索过程可能将敏感文档暴露给无权限用户」的 5 个解决方案进行独立可行性分析。

---

## 方案一：检索层访问控制（Retrieval-Time Access Control）

### 技术可行性
**实现难度：低**

主流向量数据库（Pinecone、Milvus、Weaviate、Supabase pgvector、Qdrant）均已原生支持元数据过滤功能，无需对向量检索算法做任何修改。开发者只需在索引阶段为每个 chunk 附加权限字段，在查询时传入当前用户的角色/ID 作为过滤条件即可。与现有认证体系（JWT、OAuth2.0）的集成也有成熟路径。

### 资源需求
- **算力**：可忽略，元数据过滤本质是索引扫描，开销极小。
- **数据**：需要在现有文档索引流程中增加权限元数据标注。若文档数量庞大，初次入库权限标注工作量中等。
- **人力**：1-2 名工程师，1-2 周即可完成基础实现。

### 预期效果评估
能有效阻止未授权用户在检索阶段获取受保护的文档块，是最直接的防线。但对已获得合法访问权限的内部威胁（内部人员滥用）效果有限。

### 潜在风险与局限
- **权限元数据维护成本**：文档权限变更（如员工离职、角色变更）需同步更新向量数据库中所有相关 chunk 的元数据，数量大时更新一致性是挑战。
- **chunk 级粒度不足**：文档级权限容易实现，但同一文档中不同段落的差异化权限管理较复杂。
- **旁路攻击**：若攻击者可直接访问向量数据库 API（绕过应用层），过滤条件形同虚设，需额外做数据库层鉴权。

### 综合推荐指数：⭐⭐⭐⭐⭐（5/5）

**理由**：工程实现成本最低、效果直接可验证、生态支持成熟，是任何 RAG 系统的必选基础方案，应作为首要优先级实施。

---

## 方案二：细粒度授权策略引擎（Fine-Grained Authorization Engine）

### 技术可行性
**实现难度：中**

Cerbos、OPA（Open Policy Agent）、Auth0 FGA 等工具均提供完善的 SDK 和 REST API，可与 Python/Node.js RAG 管道集成。核心挑战在于将业务权限逻辑（"谁能看哪类文档"）翻译为策略规则，需要与业务方协作梳理权限矩阵。

### 资源需求
- **算力**：授权决策通常在毫秒级完成，对延迟影响极小；但大规模并发场景下授权服务需高可用部署。
- **数据**：需维护用户-角色-资源的权限关系数据库，与身份管理系统同步。
- **人力**：需要专职的安全/权限工程师 1 名；策略规则设计需要业务分析师参与；初始建设约 2-4 周。

### 预期效果评估
相比方案一，策略引擎提供了更强的灵活性：可实现属性级别（ABAC）、关系级别（ReBAC）的细粒度控制，支持动态策略更新而无需重新部署代码。在权限规则复杂多变的企业环境中（如医疗、金融）价值显著。

### 潜在风险与局限
- **引入新的依赖服务**：授权引擎本身成为新的单点故障风险，需设计降级策略。
- **策略规则维护复杂性**：随着业务增长，策略规则可能演变为难以维护的"权限迷宫"，需要配套治理流程。
- **与向量检索的集成难度**：需要将授权引擎输出的查询计划精确映射为向量数据库的过滤语法，不同数据库适配工作量不同。

### 综合推荐指数：⭐⭐⭐⭐（4/5）

**理由**：适合权限逻辑复杂、多租户、需要审计合规的大型系统；中小团队可优先用方案一满足基本需求，再按需引入策略引擎。

## 方案三：PII 检测与脱敏（PII Detection & Redaction）

### 技术可行性
**实现难度：中**

Microsoft Presidio 是目前最成熟的开源 PII 检测脱敏工具，支持 Python 集成，兼容 LlamaIndex 等主流 RAG 框架。Amazon Comprehend 提供托管服务，适合 AWS 生态用户。技术上可行，但 PII 检测的准召率取决于领域特殊性（如专业医疗术语、非标准格式的 ID 号）。

### 资源需求
- **算力**：NER 模型推理有一定计算开销（特别是深度学习模型），大批量入库时需评估吞吐量。
- **数据**：无特殊数据需求，但需要针对特定领域的 PII 类型定制检测规则（如病历号、合同编号等非标准实体）。
- **人力**：集成 Presidio 约需 1 周；自定义检测规则调优可能需要 2-4 周；需要隐私/合规顾问参与定义 PII 范围。

### 预期效果评估
对于防止个人隐私数据（姓名、电话、邮件、身份证号等）泄露非常有效，是 GDPR/CCPA 合规的重要技术手段。但脱敏后文档的语义可用性可能有所下降（尤其是高密度 PII 文档），影响 RAG 的检索和生成质量。

### 潜在风险与局限
- **漏检风险**：隐晦的 PII 表达（如绰号、代号）难以被规则系统识别，召回率存在上限。
- **语义损失**：对人名高度相关的查询（如"找到所有提到 John Smith 合同的文档"），脱敏后检索能力可能大幅下降。
- **脱敏不可逆**：若原始文档已被脱敏入库，合法用户需要原始 PII 时难以恢复，需要分层存储策略（原始库 + 脱敏库）。
- **语言覆盖**：中文 PII 检测规则相对薄弱，需要额外开发。

### 综合推荐指数：⭐⭐⭐⭐（4/5）

**理由**：是合规性要求高场景（医疗、金融、法律）的必备防护层，与方案一组合使用效果最佳。注意需针对具体业务场景调优，避免过度脱敏损害可用性。

## 方案四：差分隐私（Differential Privacy for RAG）

### 技术可行性
**实现难度：高**

差分隐私在理论上提供了严格的数学隐私保证，但在 RAG 系统中的工程化落地较为复杂。DP-RAG 框架（sarus-tech/dp-rag）提供了参考实现，但其 token-by-token 聚合机制会大幅增加推理延迟，且隐私预算（ε 值）的设定需要专业知识。目前仍处于学术研究与工程实验阶段。

### 资源需求
- **算力**：噪声注入和多文档聚合机制显著增加推理计算量，可能比标准 RAG 慢 2-5 倍。
- **数据**：无额外数据需求，但需要大量实验数据来校准隐私预算参数（ε, δ）。
- **人力**：需要具备差分隐私理论背景的算法工程师，招聘难度高；工程调优周期长（数月）。

### 预期效果评估
是五个方案中唯一能提供数学可证明隐私保证的方案，对抗成员推断攻击（membership inference attack）和数据重建攻击效果最优。但实际应用中，隐私-可用性权衡（privacy-utility tradeoff）往往导致生成质量显著下降，ε 值越小隐私越强但 RAG 答案质量越差。

### 潜在风险与局限
- **可用性损失严重**：在强隐私保证（小 ε）下，噪声会淹没有效信息，生成质量大幅下降。
- **延迟高**：不适合对响应时间敏感的在线服务场景。
- **工程成熟度低**：开源实现仍处于早期阶段，生产环境稳定性未经充分验证。
- **参数调优困难**：隐私预算设定缺乏通用标准，依赖具体业务场景实验。

### 综合推荐指数：⭐⭐（2/5）

**理由**：技术前沿但工程落地成本极高，当前阶段仅适合有强合规要求（如政府、军事）且具备算法研究能力的机构进行探索性实践；对大多数企业 RAG 应用而言，方案一 + 方案三 的组合已能满足需求。

## 方案五：加密检索与可证明安全 RAG（Provably Secure RAG）

### 技术可行性
**实现难度：高**

SAG（Secure Augmented Generation）等框架在学术论文中已证明可行性，但将加密机制融入向量检索的工程实现目前仍处于早期阶段。传统向量相似度计算（如余弦相似度）在加密域中的高效实现（同态加密路径）计算开销极大，距离生产可用仍有较大工程差距。

### 资源需求
- **算力**：加密/解密操作对检索吞吐量影响显著；同态加密路径的计算量可能比明文检索高出数百倍。
- **数据**：需要维护加密密钥管理基础设施（KMS），增加运维复杂度。
- **人力**：需要密码学专业人才；系统集成复杂，预估建设周期半年以上。

### 预期效果评估
从根本上解决了向量数据库本身被攻破后的数据安全问题（即使数据库遭到入侵，攻击者也无法还原明文），是最彻底的安全方案。但目前工程实现的性能代价过高，限制了实际应用场景。

### 潜在风险与局限
- **性能瓶颈**：加密检索延迟可能从毫秒级上升至秒级，难以满足实时应用需求。
- **生态不成熟**：与现有向量数据库（Pinecone、Milvus 等）的集成缺乏成熟的标准和工具链。
- **密钥管理复杂**：加密密钥的安全存储、轮换、恢复机制本身带来新的安全挑战。
- **研究阶段为主**：SAG 等框架尚未经过大规模生产环境验证。

### 综合推荐指数：⭐⭐（2/5）

**理由**：技术路线方向正确但工程成熟度不足，适合作为长期技术储备方向跟踪研究，当前不建议作为主要防护手段在生产环境部署。

## 最终推荐方案及实施路径

### 立即实施（第一优先级）：方案一 + 方案三 组合

**推荐理由**：

方案一（检索层访问控制）是防止数据泄露的核心防线，实现成本最低、效果最直接，与现有主流向量数据库深度集成，应作为所有 RAG 系统的**基础安全要求**。

方案三（PII 脱敏）是合规性的重要补充，尤其在涉及 GDPR、HIPAA 等法规的场景中不可或缺。与方案一互补：方案一控制"谁能看到什么文档"，方案三控制"即使能看到文档，敏感字段也已脱敏"。

### 中期规划（第二优先级）：方案二

当系统规模扩大、权限逻辑变复杂时，引入 Cerbos 或 Auth0 FGA 等专业授权引擎，将权限逻辑从业务代码解耦，提升可维护性和审计能力。

### 长期跟踪（研究储备）：方案四 + 方案五

对于有极强合规需求（政府、军事、金融监管）的机构，可跟踪 DP-RAG 和加密检索的工程化进展，待技术成熟后考虑引入。当前不建议在生产环境中作为主要方案部署。
