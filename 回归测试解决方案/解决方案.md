# RAG 回归测试困难：修复一个问题容易引入新的退化

## 问题描述（原文）

> 回归测试困难——修复一个问题容易引入新的退化

在 RAG 系统中，每次对检索器、分块策略、Embedding 模型、Prompt 模板或生成模型进行调整时，都存在"顾此失彼"的风险：改善了某类问题的同时，可能悄无声息地在其他维度引入退化。由于 RAG 的评估链路长（检索→重排→生成），且影响因素复杂，传统软件的单元回归测试难以直接迁移，导致问题往往在上线后才被用户反馈发现。

---

## 解决方案

### 方案一：黄金数据集（Golden Dataset）+ 持续评估基线

**核心思路**

建立一个高质量、人工验证的"黄金数据集"作为稳定基线，每次变更后均与该基线对比，自动检测退化。

**实现方法**

1. 由领域专家精心标注 50～200 个代表性查询，每条包含：查询语句、期望检索文档、期望答案。
2. 将黄金数据集版本化管理（存入 Git），与代码同步演进。
3. 每次 PR/commit 触发自动评估脚本，计算与基线的 Recall@K、MRR、faithfulness、answer relevance 等指标差值。
4. 设定阈值（如各指标退化不超过 2%），超出则自动阻断合并。

**推荐工具/框架**

- [RAGAS](https://github.com/explodinggradients/ragas)：专为 RAG 评估设计，支持自动化评分
- [DeepEval](https://github.com/confident-ai/deepeval)：Pytest 风格的 RAG 单元测试框架
- [Braintrust](https://www.braintrust.dev/)：实验追踪与基线对比平台

**参考来源**

- [Golden Datasets for GenAI Testing – Techment](https://www.techment.com/blogs/golden-datasets-for-genai-testing/)
- [The Path to a Golden Dataset – Microsoft Data Science Blog (Medium)](https://medium.com/data-science-at-microsoft/the-path-to-a-golden-dataset-or-how-to-evaluate-your-rag-045e23d1f13f)

### 方案二：RAG 指标集成到 CI/CD 流水线

**核心思路**

将 RAG 质量评估嵌入 DevOps 流水线，使每一次代码变更都自动触发评估，退化行为在进入生产前被拦截。

**实现方法**

1. 在 GitHub Actions / CircleCI / Jenkins 等 CI 系统中添加评估 step。
2. 评估脚本运行黄金数据集上的 RAG 推理，输出多维指标报告。
3. 设置质量门控（Quality Gate）：context relevance、faithfulness、answer relevance 均须达标。
4. 评估报告自动附到 PR 评论，供 reviewer 参考。

**实现示例（DeepEval + GitHub Actions）**

```yaml
- name: RAG Regression Test
  run: |
    pip install deepeval ragas
    deepeval test run tests/test_rag_regression.py \
      --min-success-rate 0.85
```

### 方案三：LLM-as-a-Judge 自动评分

**核心思路**

利用强大 LLM（如 GPT-4o、Claude）充当评判者，自动为 RAG 输出打分，替代人工标注，实现大规模回归检测。

**实现方法**

1. 对每个测试用例，让 LLM 评判：检索到的上下文是否相关？生成答案是否忠实于上下文？答案是否回答了问题？
2. 使用结构化 Prompt 输出 JSON 格式评分（0～1）和理由，确保可解释性。
3. 汇总多个维度的分数，生成回归报告，自动与历史基线比较。
4. 对低分样本进行错误案例归档，供后续分析。

**三大核心指标（RAG Triad）**

| 指标 | 含义 | 评判角度 |
|------|------|----------|
| Context Relevance | 检索内容与问题相关性 | 检索端质量 |
| Faithfulness | 答案是否忠实于检索内容 | 生成端幻觉控制 |
| Answer Relevance | 答案是否回答了原始问题 | 端到端效果 |

- [DeepEval RAG Evaluation Guide](https://deepeval.com/guides/guides-rag-evaluation)

### 方案四：分层回归测试架构（Unit → Integration → E2E）

**核心思路**

仿照传统软件的测试金字塔，将 RAG 回归测试分为三层，各层独立可测，逐层定位退化根因。

**实现方法**

- **单元层（Unit Test）**：独立测试各组件（分块函数、Embedding 模型输出、向量检索 Top-K 结果），确保组件行为稳定。
- **集成层（Integration Test）**：测试检索器与 Reranker 的联合效果，使用固定查询集验证排名一致性。
- **端到端层（E2E Test）**：使用黄金数据集验证最终生成质量，包含 faithfulness、citation accuracy 等综合指标。

**分层隔离的好处**：当 E2E 测试退化时，可快速定位是哪一层引入了问题，避免"全链路排查"的高成本。

### 方案五：金丝雀部署（Canary Release）+ 影子测试（Shadow Testing）

**核心思路**

变更上线时不直接全量替换，而是将新版本以金丝雀方式逐步引入真实流量，并利用影子模式在不影响用户的情况下对比新旧系统差异。

**实现方法**

1. **金丝雀发布**：新版本先接收 5%～10% 真实流量，对比新旧版本的实时质量指标（延迟、用户满意度、幻觉率）。
2. **影子测试**：将线上请求复制一份并发给新版本，两个版本独立响应，但只有旧版本结果返回给用户；后台比对两者差异。
3. **自动晋级/回滚**：若金丝雀阶段新版本质量优于或等于基线，则自动扩大流量至 100%；否则触发回滚。
4. 配合特征开关（Feature Flag）实现无停机回滚。
