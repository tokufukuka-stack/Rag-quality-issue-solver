# 可行性分析：RAG 中"看起来正确"的幻觉自动检测

---

## 方案一：基于声明分解的事实核查（Claim Decomposition + NLI）

### 技术可行性
**实现难度：中**

声明分解依赖 LLM 对答案的语义解析，NLI 模型已有成熟的预训练版本可直接调用。核心挑战在于声明分解质量的稳定性（LLM 可能遗漏或合并部分声明），以及 NLI 模型对跨领域专业文本的泛化能力。整体技术栈成熟，接入 RAGAS 或 FActScore 框架几天内可完成 PoC。

### 资源需求
- **算力**：声明分解调用 LLM API（中等费用），NLI 推理可在 CPU 或低配 GPU 上完成。
- **数据**：无需额外标注训练集（使用预训练 NLI 模型）。
- **人力**：1 名工程师，约 1-2 周完成集成。

### 预期效果评估
- 对"部分正确但局部幻觉"类型的检测效果优秀。
- 在专业领域（医疗、法律、金融），NLI 模型的召回率可能不稳定。
- RAGAS 基准显示：在 PubMedQA 等问答数据集上 Faithfulness 指标表现良好，但对复杂推理类问题覆盖有限。

### 潜在风险与局限
- 声明分解本身可能出错（声明遗漏、过度分解），导致评分偏差。
- NLI 模型在隐含语义矛盾（非显性矛盾）场景下准确率下降。
- 成本随 token 数线性增长，高流量场景成本较高。

### 综合推荐指数：⭐⭐⭐⭐（4/5）

## 方案二：SelfCheckGPT——多次采样一致性检验

### 技术可行性
**实现难度：低**

该方法无需额外训练，只需多次调用已有 LLM 并计算语义一致性，实现极为简单。开源库已封装完整管道，代码层面数小时可完成接入。

### 资源需求
- **算力**：每次检测需 N（通常 5-10）次 LLM 调用，API 成本是单次调用的 5-10 倍。
- **数据**：无需标注数据。
- **人力**：0.5 名工程师，约 1 周完成集成与调优。

### 预期效果评估
- 对"模型捏造事实"类幻觉效果较好（不同采样中幻觉细节不稳定）。
- 对"一致性错误"类幻觉效果有限——若模型在某话题上有系统性偏见，多次采样会给出相同的幻觉，导致漏检。
- 延迟显著增加（5-10 倍），不适合实时对话场景。

### 潜在风险与局限
- **最大局限**：系统性幻觉（即模型始终"相信"的错误信息）无法被此方法检测。
- API 成本大幅增加，不适合高并发生产环境的在线评估。
- 结果对 temperature 参数较敏感，需仔细调优。

### 综合推荐指数：⭐⭐⭐（3/5）

适合离线评估、定期质检，不适合实时在线检测。

## 方案三：LLM-as-Judge 评估管道

### 技术可行性
**实现难度：低~中**

基于现有强力 LLM（GPT-4o、Claude 3.5 Sonnet）设计评判 prompt 即可快速上线，DeepEval、UpTrain 等框架已有开箱即用的实现。挑战在于 Judge LLM 本身的可靠性（Judge 也可能出错）和 prompt 的鲁棒性设计。

### 资源需求
- **算力**：Judge 调用一次高性能 LLM API，成本中等；若使用较小的蒸馏 Judge 模型（如 Llama-3-8B 微调版），可大幅降低成本。
- **数据**：设计评判 prompt 需要少量标注数据验证效果。
- **人力**：1 名工程师，约 1-2 周完成设计与验证。

### 预期效果评估
- 对复杂推理类幻觉的识别能力优于 NLI 方法（因为 Judge LLM 具备推理能力）。
- 多模型集成（Ensemble）可进一步提升鲁棒性，但成本倍增。
- 在 FaithBench、RAGTruth 等基准上，LLM Judge 整体表现超过传统 NLI 方法。

### 潜在风险与局限
- Judge LLM 本身存在偏见（如偏好更长或更自信的回答）。
- "Judge 幻觉"问题：Judge 也可能对幻觉内容给出虚假的高分。
- 延迟和成本高于 NLI 方案，大规模部署需要成本控制策略。
- 不同 Judge 模型间结果一致性存在差异，可复现性较差。

### 综合推荐指数：⭐⭐⭐⭐（4/5）

推荐用于对质量要求高、可接受一定延迟的场景（如异步批量评估、高价值 query 的在线评估）。

## 方案四：MetaRAG——变形测试（Metamorphic Testing）

### 技术可行性
**实现难度：中~高**

变形测试框架的设计需要对目标任务有深刻理解，不同领域的"变形规则"各不相同，需要手动设计或半自动生成。已有 MetaRAG 论文实现作为参考，但落地到特定业务场景需要大量定制化工作。

### 资源需求
- **算力**：每次检测需要多次 LLM 调用（生成变形 query + 收集回答），成本中等。
- **数据**：需要为具体领域设计变形规则，可能需要少量领域专家参与。
- **人力**：2 名工程师 + 1 名领域专家，约 3-5 周完成定制实现。

### 预期效果评估
- 对"逻辑一致性幻觉"（如模型对相关问题给出矛盾回答）的检测有独特优势。
- 可发现其他方法难以捕捉的系统性错误模式。
- 对"事实细节级别"的幻觉（如数字错误）检测能力有限。

### 潜在风险与局限
- 变形规则设计质量直接决定检测效果，规则覆盖不全会导致漏检。
- 对闭域 RAG（文档范围明确）效果好，对开放域效果一般。
- 实现复杂度高，工程维护成本大。
- 变形后问题的答案可能也不在检索语料中，引入误报。

### 综合推荐指数：⭐⭐⭐（3/5）

适合需要系统性验证 RAG 鲁棒性的评估阶段，不适合生产实时检测。

## 方案五：内部激活探测（Activation Probing / Internal Confidence）

### 技术可行性
**实现难度：高**

需要访问 LLM 的内部激活层（仅适用于开源模型或可部署的私有模型），不适用于闭源 API（如 GPT-4、Claude）。训练探测分类器需要标注的幻觉数据集，构建成本较高。技术前沿性强，工程化难度大。

### 资源需求
- **算力**：需要部署完整的开源 LLM（如 Llama-3-70B），推理时提取中间层需要额外内存（约 10-20% 开销）。
- **数据**：需要幻觉/非幻觉标注数据集（数千至数万条），构建成本高。
- **人力**：2 名 ML 工程师 + 研究背景，约 6-12 周完成研究与工程化。

### 预期效果评估
- 理论上可以捕捉到其他方法漏检的"语义层面高度相似"的幻觉（即"语义幻觉"）。
- 研究结果显示在特定数据集上 AUROC 可达 0.85 以上，优于 NLI 和采样方法。
- 实时推理开销小（探测器是轻量级的）。

### 潜在风险与局限
- **严重限制**：仅适用于可获取内部激活的模型，完全不适用于闭源 API。
- 探测器的泛化能力有限，在训练集分布外的场景性能可能大幅下降。
- 标注数据集构建是最大瓶颈。
- 模型版本更新后探测器需要重新训练。

### 综合推荐指数：⭐⭐（2/5）

技术前沿，但工程落地成本极高，建议仅在开源模型部署环境且有足够研究资源时考虑。

| 方案 | 实现难度 | 延迟影响 | 检测效果 | 适用场景 | 推荐指数 |
|------|---------|---------|---------|---------|---------|
| 声明分解 + NLI | 中 | 中（+50-200ms） | ⭐⭐⭐⭐ | 生产在线/离线均可 | ⭐⭐⭐⭐ |
| SelfCheckGPT | 低 | 高（5-10倍延迟） | ⭐⭐⭐ | 离线质检 | ⭐⭐⭐ |

## 最终推荐方案及理由

**推荐采用"方案一（声明分解 + NLI）"与"方案三（LLM-as-Judge）"的分层组合策略**：

**第一层（在线实时检测）**：使用 RAGAS Faithfulness 指标（声明分解 + NLI）对所有回答进行快速评分，低于阈值的自动标记或降级处理。该层延迟可控、成本低，适合高并发场景。

**第二层（离线/异步精评）**：对标记为风险的回答，或对高价值/敏感 query，使用 LLM-as-Judge（如 GPT-4o 或 Claude）进行深度分析，给出详细原因和修正建议。

**理由**：
- 两种方法的漏检模式互补——NLI 对显性矛盾敏感，LLM Judge 对隐含推理错误更有优势。
- 成本分层合理：高流量的普通 query 用低成本 NLI，低频高价值 query 用高成本 Judge。
- 均有成熟开源工具支持（RAGAS + DeepEval），工程落地周期短（约 2-4 周）。
- 建议在 RAGAS 的 `faithfulness` 和 `answer_relevancy` 指标基础上，逐步引入自定义领域 NLI 模型以提升专业场景效果。
