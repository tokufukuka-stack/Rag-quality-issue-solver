# GDPR 数据删除权在向量库中的执行解决方案

## 问题描述（原文）

> 合规性挑战——GDPR 等法规要求的数据删除权难以在向量库中执行

GDPR 第 17 条（被遗忘权）要求数据控制者在特定情况下（数据不再必要、同意被撤回、非法处理等）必须删除用户个人数据。然而，在 RAG 系统中，个人数据经过 Embedding 模型转化为高维向量后分散存储在向量库中，存在以下技术难点：
- 向量数据与原始文本之间的映射关系不透明
- 大多数 ANN 索引（如 HNSW）采用懒删除机制，逻辑删除后物理数据仍存在
- 数据经过 chunking 后分散为多个片段，难以追踪全量关联
- 备份和副本中的数据同样需要删除，操作复杂
- 删除后需提供合规证明，审计追踪困难

---

## 主要解决方案

### 方案一：基于文档 ID 的精准追踪删除（Document ID Lineage Tracking）

**核心思路：**
在向量入库时，为每个 chunk 附加来源文档 ID、chunk ID 以及数据主体标识符（如用户 ID）作为元数据。收到删除请求时，通过元数据过滤快速定位所有关联 chunk，执行物理删除并触发索引压缩（compaction）。

**实现方法：**
1. 建立文档 ID → chunk ID 映射表（关系数据库辅助存储）
2. 向量插入时携带 `{"doc_id": "xxx", "subject_id": "user_123"}` 元数据
3. 接到 GDPR 删除请求 → 查询映射表 → 调用向量库 delete-by-filter API
4. 对使用 HNSW 索引的库（如 Weaviate、Qdrant）触发强制 compaction 以物理清除节点
5. 删除源文档，重新同步知识库
6. 在审计日志中记录删除时间戳、操作人员、受影响 chunk 数量

**推荐工具/框架：**
- Weaviate（支持 delete-by-filter + GDPR batch deletion）
- Qdrant（支持 payload filter 删除）
- AWS Bedrock Knowledge Bases（自动关联 S3 源文件删除）
- LlamaIndex（Document Management 模块支持文档级更新/删除）

### 方案二：加密擦除（Cryptographic Erasure / Cryptographic Zeroization）

**核心思路：**
不直接删除向量数据，而是使用每位用户（或每份文档）独立的加密密钥对数据加密存储。需要执行"删除"时，仅销毁对应的加密密钥，使密文成为无法解密的随机噪声，从而在计算意义上等同于数据被彻底删除。

**实现方法：**
1. 数据入库前，对每位数据主体的相关文本使用其专属 AES-256 密钥加密
2. 密钥集中存储于 KMS（Key Management Service，如 AWS KMS、HashiCorp Vault）
3. 收到删除请求 → 在 KMS 中销毁对应密钥
4. 向量库中的密文向量保留但永久不可解密
5. 向监管机构证明：密钥销毁 = 数据不可访问 = 合规删除

**推荐工具/框架：**
- AWS KMS / Google Cloud KMS（云端密钥管理）
- HashiCorp Vault（自托管密钥管理）
- IronCore Labs（专为向量数据库设计的加密方案）
- [GDPR, Data Encryption, and Right of Erasure 指南](https://info.cgcompliance.com/blog/gdpr-encryption-and-right-of-erasure)

### 方案三：机器遗忘（Machine Unlearning for RAG）

**核心思路：**
将 RAG 的数据删除问题分为两层：（1）向量库层——删除对应 embedding chunk（即时、零计算成本）；（2）基础模型层——若 LLM 经过微调包含了个人数据，需通过机器遗忘技术消除模型权重中的影响。对于纯 RAG（不含微调）场景，仅需处理第一层。

**实现方法：**
1. 建立文档 → chunk → embedding 的完整谱系追踪
2. 收到删除请求 → 从向量库删除所有关联 chunk（API 调用，毫秒级）
3. 若模型经过微调：使用梯度上升（Gradient Ascent）或 SISA Training 进行机器遗忘
4. 删除缓存中的相关查询结果（cache invalidation）
5. 向 Pathway 等实时 RAG 框架注册变更，触发实时索引更新

**推荐工具/框架：**
- Pathway（实时 RAG + 机器遗忘支持）
- [When Machine Unlearning Meets RAG (Arxiv 2024)](https://arxiv.org/html/2410.15267v1)
- LangChain Document Store（支持 delete + re-index）

### 方案四：数据架构隔离设计（Privacy-by-Design 架构）

**核心思路：**
从系统设计阶段即将 PII（个人身份信息）与非 PII 数据分离存储。向量库中只存储去标识化（de-identified）的内容向量，原始 PII 保存在独立的关系型数据库中，两者通过 token 关联。删除时只需从关系数据库中删除 PII 记录，向量库中的内容无法反向追溯到个人。

**实现方法：**
1. 入库前进行 PII 检测（使用 Presidio、spaCy NER 等工具）
2. 替换或屏蔽 PII 字段后再进行 Embedding
3. 关系数据库存储：`{token_id, user_id, original_pii_data}`
4. 向量库存储：`{token_id, vector, de-identified_text}`
5. 删除请求 → 仅删除关系数据库中的 PII 记录 → 向量库记录因无个人关联自动符合 GDPR

**推荐工具/框架：**
- Microsoft Presidio（开源 PII 检测与匿名化）
- AWS Comprehend（PII 实体检测）
- spaCy（NER + 自定义实体识别）
- Privacera（向量数据库安全与隐私管控平台）

### 方案五：合规工作流 + 自动化审计追踪

**核心思路：**
构建端到端的合规工作流，包括：GDPR 删除请求接收 → 数据定位 → 执行删除 → 验证删除 → 生成合规证明报告，全程自动化并留存审计记录。

**实现方法：**
1. 建立 DSAR（Data Subject Access Request）处理系统，自动触发删除流程
2. 构建跨组件数据谱系图（Data Lineage Graph）：源文档 → chunks → 向量 → 缓存
3. 使用 Apache Atlas 或 DataHub 进行数据目录管理和谱系追踪
4. 删除完成后生成可审计的合规报告（含时间戳、操作记录、受影响数据量）
5. 定期全量扫描验证已删除数据不可访问

**推荐工具/框架：**
- Apache Atlas（数据治理与谱系追踪）
- DataHub（开源数据目录）
- OneTrust / TrustArc（GDPR 合规管理平台）
- Collibra（企业数据治理）
